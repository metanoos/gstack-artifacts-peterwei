## Executive Verdict

Build the multi-AI task and correspondence model now, while running the writing-audition evidence ladder in parallel. Zine is pre-production, so this should be one clean protocol and application cut: no compatibility aliases, dual reads, migration flags, or legacy UI. The product thesis is broader than an audition feature: Zine becomes the local-first durable press in which a human can write, converse with several AI harnesses, move selected context between them, and preserve an inspectable account of what each participant saw and contributed.

The core constraint is epistemic honesty. Zine can prove what entered its own signed trace and what an integrated adapter observed. It cannot prove that no outside AI was used, reconstruct hidden provider reasoning, or turn behavioral timing into authorship proof. The UI and protocol must preserve that distinction.

The discovery gate from the source design remains valuable evidence work, but the user explicitly chose implementation in parallel rather than making Gate 1 a feature-code prerequisite.

## Vision

### 10x Check

The ordinary version is an editor with a model button and a provenance sidebar. The 10x version is a local-first correspondence press:

- a human can conduct one cumulative task with GLM, Codex, Claude, or a local model;
- every accepted human or AI turn becomes an ordinary, readable Zine Step;
- the same interaction can branch into independently owned reply files, like letters;
- exact context, prompt rules, provider identity, approvals, retries, and outcomes are inspectable without polluting the authored prose;
- source quotation, model output, external paste, and manual writing remain visually and durably distinguishable;
- a reviewer can inspect the raw replay, then perform a separate human-AI analysis whose own provenance is recorded;
- the durable task belongs to the user and remains useful after any provider session or subscription ends.

This makes switching models a change of correspondent, not a loss of history.

### Platonic Ideal

Zine should feel like a writing desk with a perfect archive. The active page is always understandable as a page. The archive can answer, without guesswork:

1. What exact artifact state did this participant see?
2. Which exact sources and prompt rules were supplied?
3. What did the participant return, and what was accepted?
4. Was this a continuation, a reply, a transformation, an analysis, or a compaction?
5. Which statements came from manual authorship, a named model session, the filesystem, or an external paste?
6. Which relationships are editorial citations, conversational replies, task context, or later analysis evidence?

The ideal is not surveillance. It is voluntary, high-fidelity recordkeeping with explicit gaps.

## Product Principles

1. **Ordinary files first.** Tasks, replies, reports, and analyses are normal `z:file` traces, not a parallel chat database.
2. **Every accepted turn remains readable in history.** Before compaction, a task snapshot contains the complete thread to that Step. After explicit compaction, it contains the complete active transcript plus signed references to every covered turn; replay through `prev` still exposes the verbatim originals.
3. **No invisible mutation.** Context files never auto-Step. Failed, rejected, cancelled, or preview-only model operations never create artifact Steps.
4. **Context is disclosed, not inherited.** LLM-context relationships belong to one operation and remain in that historical Step; they do not stick to the next Step.
5. **Relationships have roles.** A bare `q` edge is insufficient for mixed reply, quotation, context, tag, and analysis semantics.
6. **Provider claims are bounded.** Zine records observable messages, tool events, approvals, and supplied context. It does not claim hidden prompts, chain of thought, or provider-side completeness.
7. **Local-first ownership.** Credentials stay in the client, self-hosting remains available to capable agencies, and provider subscriptions are adapters rather than the archive.
8. **One clean pre-production model.** Remove superseded names and fields outright.

## Authoritative Data Model

### Normative schema cut

The protocol specification, not this plan, will own the final RFC-2119 language. T1 must begin by landing the following concrete shape and golden fixtures before application behavior. All new `TraceNode` content uses `schema: 2`; pre-production schema-1 task/LLM forms are not read or translated.

Every `TraceNode` remains a NIP-01 event whose ID and signature cover the exact UTF-8 `content` string and tags. Hash fields are lowercase SHA-256 hex. Request hashes cover the exact adapter request bytes and record their media type plus encoder version. Structured manifest hashes use RFC 8785 JSON Canonicalization Scheme UTF-8 bytes. Event content itself follows the existing compact-JSON convention; golden fixtures pin exact bytes, event IDs, and signatures.

An abbreviated valid file node is:

```json
{
  "schema": 2,
  "snapshot": "…complete active file text…",
  "contentHash": "<64 lowercase hex>",
  "relations": [
    {
      "qIndex": 0,
      "target": "<event id>",
      "roles": ["analysis-evidence", "inline"],
      "occurrences": [{"kind": "span", "start": 12, "end": 38}]
    }
  ],
  "injectRules": ["<rule Step id>", "<rule Step id>"],
  "task": {
    "version": 1,
    "root": "<task genesis id; omitted on genesis>",
    "turns": [
      {
        "id": "<32 lowercase random hex>",
        "index": 0,
        "kind": "message",
        "actor": {"origin": "author", "voice": "<pubkey>"},
        "range": {"start": 0, "end": 128},
        "action": "author",
        "contextStepIds": [],
        "injectRuleStepIds": []
      }
    ]
  },
  "operation": {
    "attemptId": "<32 lowercase random hex>",
    "contract": "continue",
    "requestHash": "<64 lowercase hex>",
    "requestMediaType": "application/json",
    "requestEncoder": "glm-json-v1",
    "preparedHead": "<event id>",
    "resultHash": "<64 lowercase hex>"
  }
}
```

Validation rules:

- `schema`, `snapshot`, and `contentHash` are required; the hash must match the snapshot under existing file rules.
- `relations` is required whenever a `q` tag exists. Its length equals the `q`-tag count. `qIndex` is the zero-based ordinal among q tags only—not the absolute index in the event's mixed tag array—with ordinals 0 through q-count−1 present exactly once; `target` must equal that q tag's event ID. A golden fixture interleaves `z`, `f`, `e`, `q`, `t`, and a second `q` and binds relation ordinals 0 and 1.
- `roles` is a non-empty, sorted, duplicate-free set. Multiple roles are allowed on one edge. This is how an analysis quotation can be both `analysis-evidence` and `inline` without duplicate q tags.
- Initial roles are `member`, `inline`, `tag`, `reply`, `llm-context`, `analysis-subject`, `analysis-evidence`, `handoff-context`, and `compact-source`.
- Role-specific qualifiers are optional but typed: `occurrences` for UTF-16 spans, `replyTurnId` for a turn anchor, and `packetEntry` for a consent-manifest entry. Unknown qualifier keys are invalid for schema 2 writers.
- A reader encountering an unknown future role preserves and displays the signed edge as unsupported, but MUST NOT place it in citation, reply, context, membership, or ranking indexes.
- `injectRules` is ordered, duplicate-free, and contains exact immutable rule Step IDs. The singular `injectRule` key is invalid.
- `task.turns` array order is display order. Its ranges are ordered, non-overlapping UTF-16 ranges into the snapshot. `index` is an immutable creation sequence, unique within the task and assigned as `max(all active and covered indexes)+1`; after compaction, array/display order need not equal creation-index order. Turn IDs and indexes never change.
- `operation` appears only on an artifact Step created by an accepted model/agent mutation and binds that Step to one signed attempt summary.

The complete closed Turn shape is:

```json
{
  "id": "<32 lowercase random hex>",
  "index": 7,
  "kind": "message|compact",
  "actor": {
    "origin": "author|model|external|filesystem",
    "voice": "<pubkey>",
    "provider": "<adapter id; model only>",
    "model": "<model id; model only>",
    "sessionId": "<observed provider session; model only>"
  },
  "range": {"start": 400, "end": 620},
  "action": "author|continue|reply|compact|analyze",
  "contextStepIds": ["<ordered exact Step ids>"],
  "injectRuleStepIds": ["<ordered exact rule Step ids>"],
  "attemptId": "<required for model/agent turns>",
  "requestHash": "<required for model/agent turns>",
  "sourceHead": "<prior target head; omitted on genesis>",
  "replyTo": {"stepId": "<exact source>", "turnId": "<optional exact turn>"},
  "covers": [{"turnId": "<covered id>", "turnIndex": 2, "sourceStepId": "<verbatim source Step>"}]
}
```

All listed keys other than the explicitly conditional keys are required, including empty context/rule arrays. `replyTo` exists only for the genesis turn of a reply file. `covers` exists only for `kind: compact` and contains at least one contiguous active display range in its prior Step. The resulting head is derived as the containing event ID because an event cannot include its own ID before signing; it is not serialized inside the Turn.

Normative fixture cases:

- human append: `kind:message`, `action:author`, AUTHOR actor, empty context/rules, no attempt/request fields;
- model Continue: `kind:message`, `action:continue`, MODEL actor, exact context/rules, attempt/request fields, source head;
- reply genesis: index 0, `action:reply`, `replyTo`, no `sourceHead` on the new trace; MODEL form has attempt/request fields and AUTHOR form does not;
- compact: new highest creation index, `kind:compact`, displayed at the first covered turn's former position, `covers` every removed active turn in prior display order, and preserves each covered ID/index/source Step.

A golden compacted-task fixture must show an active array such as `[turn 0, compact turn 5, turn 3, turn 4]`: display ranges increase, creation indexes intentionally do not, and the next turn receives index 6.

Golden invalid writer fixtures must cover a missing relation row, wrong `qIndex`, target mismatch, empty/duplicate/unsorted roles, writer-side unknown role, invalid range, singular `injectRule`, duplicate turn ID/index, task-root mismatch, request byte/hash mismatch, and an operation whose attempt was not eligible to apply. A separate forward-compatibility fixture proves a reader preserves a future unknown role as unsupported and unindexed.

Operational records use a new provisional regular event, `TaskOperation` kind `4295`, rather than moving the file's `prev` head. Owner-stream records target the exact task head with `e` role `task`; successive records link with `e` role `op-prev`. Its compact content is:

```json
{
  "schema": 1,
  "taskRoot": "<task genesis event id>",
  "streamId": "<owner stream or receipt stream id>",
  "seq": 12,
  "kind": "attempt|handoff|consent|transfer-ack|reconcile",
  "visibility": "local|packet-eligible",
  "record": {"<kind-specific, versioned payload>": "…"}
}
```

`TaskOperation` is append-only evidence adjacent to the ordinary file trace. It never changes the artifact head. Attempts, approvals, handoffs, and transfer receipts are therefore trace-backed without inventing empty file revisions. The exact provisional kind number and schemas land atomically with the rest of T1; there is no reader for an older operational-event form.

Signer and chain rules are explicit:

- `attempt`, `handoff`, `consent`, and `reconcile` records belong to `streamId = owner:<taskRoot>:<ownerPubkey>` and MUST be signed by the file trace owner.
- The owner stream is serialized by a per-task operation lock plus compare-and-set on its current operation head. `seq` starts at 0, increments by one, and every non-genesis record has exactly one `op-prev` to seq-1.
- The latest state for an attempt is the highest contiguous owner-stream seq carrying that attempt ID. A fork—two records with the same `op-prev`/seq—makes the stream ambiguous and prevents dispatch, handoff, completeness claims, or artifact apply.
- The owner resolves a fork with `kind:reconcile`. `seq = max(all fork-tip seq)+1`; its sole structural `op-prev` MUST target the lexicographically least maximum-sequence tip so the predecessor is always seq−1, and `op-merge` tags name every other tip. The payload's `chosenTipId` separately states which branch's logical state wins and may name any tip; `discardedTipIds` names the rest. Readers accept reconciliation only when the same owner signed every branch and the reconcile record. Records descending from an already visible unresolved fork are ineligible to advance state, so reconciliation is the first valid post-fork record.
- `transfer-ack` belongs to `streamId = receipt:<manifestHash>:<receiverPubkey>`, targets the exact consent event with `e` role `consent`, and MUST be signed by the named destination/receiver key. It never joins or advances the owner's operation stream. Receipt seq normally starts at 0; later delivery-status receipts chain within that receipt stream only.

Kind-specific payloads are closed versioned objects:

- `attempt`: `{version, attemptId, retryOf?, targetHead, targetHash, contract, provider:{adapter,model,sessionId?,requestId?}, request:{hash,mediaType,encoder,orderedContextStepIds[],orderedInjectRuleStepIds[],capabilities[]}, execution, application, approvals[], result?:{hash,mediaType,observableMessage?}, linkedStepId?, lateEvents[]}`.
- `handoff`: `{version, sourceHead, orderedTurnIds[], orderedArtifactStepIds[], orderedContextStepIds[], orderedInjectRuleStepIds[], orderedUnresolvedAttemptIds[], orderedToolRefs[], destination:{adapter,model}, renderedRequestHash, manifestHash, providerSessionRef?}`.
- `consent`: the transfer manifest defined under Consent packet semantics.
- `transfer-ack`: `{version, manifestHash, receivedEventIds[], historyComplete, receivedAt}`.
- `reconcile`: `{version, chosenTipId, discardedTipIds[], reason}`.

Arrays whose name begins `ordered` preserve user-reviewed order; all other ID arrays are sorted lexicographically for canonical hashing. Prompt-rule and context arrays are always order-preserving and therefore always use the `ordered` prefix. Required keys may not be null. Optional keys are omitted, not null. Unknown keys and enum values are invalid for schema-1 writers; readers preserve the signed raw event, mark it unsupported, and do not use it to authorize dispatch, apply, handoff, or completeness claims. Golden valid examples prove that swapping context/rule order changes request/manifest hashes and pin request bytes, manifest canonical bytes, event IDs, and signatures; golden invalid examples exercise every required field, enum, state transition, cross-link, and hash mismatch.

### 1. File and Step

Every artifact remains an ordinary signed file trace. `prev` is the revision chain. A task file's content is the complete active readable transcript as of that Step; covered pre-compaction turns remain verbatim in prior Steps and addressable through compaction metadata. A reply file begins its own trace and carries one durable reply relation to the exact source Step.

### 2. Typed relationships

`q` remains the common graph edge. Each event containing `q` must carry the signed `relations` manifest above. A relation has one or more semantic roles; roles are not mutually exclusive.

Required initial roles:

- `reply`: this file/turn responds to an exact Step;
- `inline`: authored quotation or citation included in the artifact;
- `tag`: organizational association;
- `llm-context`: content supplied to one model operation;
- `analysis-evidence`: material the analysis relied on;
- `analysis-subject`: the trace or packet being analyzed;
- `handoff-context`: exact content carried to another provider session;
- `compact-source`: historical turn range transformed by explicit compaction.
- `member`: folder composition, preserving the current q/member ordering invariant.

Readers classify per relation and per role. `inline` and `tag` feed citation/composition indexes; `reply` feeds correspondence; `member` feeds folder membership; `analysis-evidence` feeds evidence views; and `llm-context`/`handoff-context` feed “Context used.” Context-only roles are excluded from cited-by, citation totals, and ranking. When an edge has both `inline` and `analysis-evidence`, it appears once in citation indexes and once in evidence views.

### 3. Task turns

Each accepted append has durable metadata:

- `taskId` owned by the Zine task trace;
- monotonically increasing `turnIndex` within the task;
- `turnId` stable across rendering;
- actor origin and contributor attribution;
- action contract and optional provider session;
- exact ordered context Step IDs and prompt-rule Step IDs;
- attempt ID when produced through a model;
- request fingerprint;
- source head; the resulting head is the containing event ID and therefore derived after signing.

One external provider session maps to one Zine task. One Zine task may attach multiple provider sessions sequentially. The task trace, not the provider session, is canonical.

Task formation is explicit:

- `Start Task` on an empty file creates genesis with an empty turn list; the first append-and-Step creates turn index 0. On a non-empty ordinary file, conversion previews one initial turn covering the existing snapshot and Steps a new schema-2 task node.
- Task mode is append-only at the authored-text layer. The composer appends exactly one turn per Step. Direct edits or deletion inside an earlier turn are blocked; correction is a new turn, a fork, or explicit compaction.
- A paste inside the composer is part of that one new turn. Its Runs may be AUTHOR, MODEL, or EXTERNAL, but pasted delimiter-like text never creates extra turn boundaries.
- `transform-selection` and `transform-file` cannot target an existing task transcript. They must create a fork or use the explicit `compact` contract.
- AUTHOR Reply creates a new ordinary file/task identity. Before its first Step it is only a draft. The first Step is that reply trace's genesis, turn index 0, with a `reply` relation to the exact source Step/optional source turn. MODEL Reply creates the same genesis atomically on acceptance.
- Continuing a reply file appends turn index 1 and later; it never inherits the source file's task root. Correspondence connects independently owned task/file roots.
- On task genesis, `task.root` is absent and readers derive it from that event's ID. Every later task Step must name the same genesis ID.

### 4. Attempts

Every dispatch creates durable `TaskOperation` checkpoints plus a local adapter journal. Attempt state has two orthogonal axes; there is no ambiguous generic `completed` state:

```text
execution:   prepared -> dispatch-intent -> dispatched -> streaming -> succeeded
                    |              |           |          |         ├-> failed
                    v              v           v          v         ├-> cancelled
                 cancelled       unknown     unknown    unknown     └-> unknown

application: prepared/dispatch-intent/dispatched/streaming/failed/cancelled/unknown -> not-eligible
             succeeded -> pending-review -> applied
                                         ├-> rejected
                                         └-> stale
```

An attempt has a stable ID, task/target, prepared head/hash, exact request hash/encoding, provider request/session identity, timestamps, observable events, approval decisions, execution state, application state, and optional `retryOf`. `application` is `not-eligible` until execution succeeds, then `pending-review`; only `pending-review` can transition to `applied`, `rejected`, or `stale`. Legal transitions are monotonic:

| From | To | Rule |
|------|----|------|
| prepared | cancelled | user cancels before provider I/O; no request was sent |
| prepared | dispatch-intent | outbox checkpoint with client request/idempotency key is durable before provider I/O |
| dispatch-intent | dispatched | provider acknowledges the exact request/session identity |
| dispatch-intent | unknown or failed | restart/reconciliation cannot prove dispatch status, or proves pre-dispatch failure |
| dispatched | streaming, succeeded, failed, cancelled, unknown | adapter evidence determines outcome |
| streaming | succeeded, failed, cancelled, unknown | disconnect without proof becomes unknown |
| unknown | succeeded or failed | provider reconciliation with the same request/session ID may resolve it; a user may mark it failed with `resolutionEvidence: user`, never succeeded |
| succeeded/pending-review | applied, rejected, stale | explicit acceptance plus CAS is required for applied |

`failed`, `cancelled`, `rejected`, `stale`, and `applied` are terminal. `prepared` may be cancelled only while no dispatch-intent exists. Once intent is durable, cancellation or ordinary redispatch is forbidden until reconciliation determines a later state. Late frames are appended as `lateEvents` and never reopen a terminal attempt or authorize apply. A rejected result can be copied manually as EXTERNAL or rerun under a new attempt, but the rejected attempt itself cannot later apply. Retries always receive new IDs and `retryOf`. Artifact application is idempotent by attempt ID. Only accepted successful mutations create an artifact Step; all intended/dispatched attempts remain inspectable through their operation chain.

Durable write order is: persist prepared attempt -> persist/fsync `dispatch-intent` with client request ID and provider idempotency key when supported -> perform provider I/O -> persist acknowledged provider request/session ID and `dispatched` -> stream journal -> persist terminal execution -> await user disposition -> CAS/apply artifact -> persist resulting Step link. An adapter capability manifest states `idempotency: native|session-reconcile|none` and `resume: true|false`. A crash after intent but before acknowledged dispatch recovers as an uncertain outbox entry, never as ordinary prepared work. On restart:

- native idempotency reissues only the provider status/reconcile call, never a new generation request;
- session reconciliation resumes/queries the exact session and message ID;
- unsupported providers with an uncertain dispatch-intent move to `unknown` and offer “inspect logs,” “mark failed,” or “start linked retry”; they never cancel as unsent or redispatch automatically;
- `unknown` is never eligible for artifact application.

### 5. Origin and contributor

The origin classes presented in the palette and replay are bounded observations, not proof of physical authorship:

- `AUTHOR / MANUAL`: editor transactions observed as direct keyboard/composition input inside Zine;
- `MODEL / LLM`: accepted output observed from a named adapter/model/session;
- `EXTERNAL / UNVERIFIED`: paste or import whose upstream source is outside the press or whose private receipt cannot be verified;
- `FILESYSTEM`: scanning/reification substrate, not an author claim.

The task owner signs the Zine event. MODEL attribution identifies a contributor observed by Zine; it is not a provider cryptographic signature. Zine does not offer a “plain manual” override for an observed paste. A user declaration may add a note, but it cannot replace EXTERNAL origin evidence.

### 6. Prompt rules

A model row stores an ordered `injectRules[]` of exact Step IDs. Rules are pinned; the row never auto-follows a newer revision. When a rule has a newer Step, the palette offers a diff and explicit update. Imported rules start disabled. Prompt rules cannot expand filesystem, network, tool, or mutation permissions.

The singular `injectRule` field is removed outright.

## Action Contracts

The registry exposes bounded contracts rather than arbitrary prompt-defined powers:

| Contract | Reads | May mutate | Artifact result |
|----------|-------|------------|-----------------|
| `continue` | focused task, selected context/rules | focused task | append one AI turn and Step |
| `reply` | exact focused Step, selected context/rules | new response file | create/Step reply atomically for MODEL; blank unstepped file for AUTHOR |
| `transform-selection` | exact selection and context | focused file | one accepted replacement Step |
| `transform-file` | exact file head and context | focused file | one accepted replacement Step |
| `create-file` | selected sources/context | new file | one new accepted file Step |
| `analyze` | subject, selected evidence/context | new analysis file by default | one analysis Step after reference validation |
| `compact` | exact contiguous task turn range | focused task | one explicit compacted Step preserving ancestry |
| `handoff` | ordered task turns, artifacts, context, rules | provider session metadata only | one signed `TaskOperation`; no artifact-head mutation |

Custom palette actions select one of these contracts and supply prompts/configuration. Prompt text never grants new mutation authority.

### Capability and output matrix

Every model row declares a subset of adapter capabilities, and every invocation shows the resolved set. Unlisted capabilities default to deny.

| Contract | Read selected Zine Steps | Read arbitrary filesystem | Network | Provider tools | Mutate focused target | Create file | Required output |
|----------|--------------------------|---------------------------|---------|----------------|-----------------------|-------------|-----------------|
| `continue` | Yes | No | adapter endpoint only | optional, approval-gated | append one turn | No | `{message}` |
| `reply` | Yes | No | adapter endpoint only | optional, approval-gated | No | one reply trace | `{message}` |
| `transform-selection` | Yes | No | adapter endpoint only | No by default | one exact range | No | `{replacement}` |
| `transform-file` | Yes | No | adapter endpoint only | No by default | one non-task file | No | `{snapshot}` |
| `create-file` | Yes | No | adapter endpoint only | optional read-only | No | one named draft/result | `{name, snapshot}` |
| `analyze` | Yes | No | adapter endpoint only | optional read-only | No | one analysis trace | `{report, evidenceRefs[]}` |
| `compact` | selected task range | No | adapter endpoint only | No | one task via special transform | No | `{summary, coveredTurnIds[]}` |
| `handoff` | exact manifest only | No | destination adapter only | No | No | No | `{providerSessionRef, receipt}` |

Filesystem/network/tool expansion is a separate provider-native approval, constrained by an adapter allowlist; a prompt rule cannot request it. For the first supported release, every contract has one primary artifact target. A provider tool that proposes another Zine mutation must return a separate typed proposal and receive its own preview, acceptance, attempt linkage, and Step. An uncontrolled process write is reconciled as FILESYSTEM/EXTERNAL evidence, never folded into the model turn. Path/name collisions block with rename/replace/cancel choices; replace still obeys prepared-head CAS. Unknown output keys are preserved in the local raw result but ignored for mutation. Missing required output fields make the result invalid and ineligible to Step. Contributor/provider/model/session attribution is copied from the prepared adapter identity and observed session, never accepted from model-controlled output; any returned contributor-like field is ignored and logged as an invalid extra.

`Receive` is removed. The existing recipe becomes `Analyze · Process`, and generated files use `-process-analysis` naming. There is no alias or migration reader.

## Core User Flows

### Continue a cumulative task

1. The user appends a human message to the task file and explicitly Steps it.
2. The user selects a model row and `Continue`.
3. Zine resolves the exact target head, ordered prompt-rule Steps, and context Steps.
4. If any referenced file has unstepped changes, the call is blocked with choices to Step manually, remove that context, or cancel.
5. Zine shows disclosure and permissions, prepares the immutable request, and records the attempt.
6. The adapter streams observable events and provider-native approval requests into Zine.
7. The user accepts the output.
8. Zine rechecks the target head/hash, appends the AI turn to the complete transcript, and creates exactly one `action:llm` Step.

### Reply as correspondence

AUTHOR Reply opens a blank unstepped response trace. Its reply relation commits with the first explicit Step. MODEL Reply prepares against the exact source Step and creates/Steps the new response file atomically on acceptance. Repeated Reply creates a graph of independently addressable letters. Continue and Reply may be mixed on every turn.

### Quote without Coin

Copying stepped Zine text records a private provenance receipt containing exact source Step and Run boundaries. Pasting into a supported editor resolves that receipt and inserts auto-bracketed `[[ quote | sourceStepId ]]` content using the source Run's voice/color, including mixed Runs. This creates an `inline` relationship. Coin remains a separate affirmative action for lines the reader thinks are great. If the receipt cannot be verified, the paste remains `EXTERNAL / UNVERIFIED`; it never becomes AUTHOR merely because the source is unresolved.

### Analyze a replay

The reviewer first sees the raw consented replay. They then create a separate agency-owned private analysis task and converse with their own AI. The resulting report is itself a Zine trace. `llm-context` records what the analysis model saw; `analysis-evidence` and `inline` record what the report relied on or quoted. Invalid or inaccessible references flag the task result and prevent an analysis artifact Step.

### Switch providers

The user starts a reviewed handoff. Zine renders the exact ordered task turn IDs, artifact/context Steps, instruction-rule Steps, unresolved attempts, and tool references to be sent. Dirty references block. The user reviews the rendered request and its fingerprint before dispatch. The receiving provider gets only this manifest. A successful handoff writes a signed `TaskOperation kind: handoff` targeting the source task head; it does not append prose or move the file head. Provider-internal compaction is recorded as an observation, never treated as canonical history.

### Compact a task

The user selects a contiguous historical turn range and previews the exact before/after. Accepted compaction creates one summary turn whose metadata contains `covers: [turnId…]`, source head, exact covered ranges, prompt-rule Steps, request hash, and `compact-source` relationships. The current snapshot is thereafter the complete **active** transcript: uncovered turns plus the summary turn. Covered turn IDs remain addressable through `{coveredTurnId, sourceStepId, compactingTurnId}` entries, and their verbatim text remains reachable through `prev`. Compaction is never automatic and never reuses or renumbers a covered turn ID.

## Security and Privacy Model

- Credentials and subscription tokens remain in the provider process or client-local secret store; never in a Step, prompt-rule artifact, attempt export, relay event, or log.
- Adapter processes receive a minimal allowlisted environment. Never inherit the whole Zine process environment.
- Executable choice is explicit. Record path/version/hash/signing identity and warn when it changes before next dispatch.
- Provider-native tool approvals are rendered without weakening. Malformed approval requests, adapter disconnects, or unknown tool schemas default to deny.
- Context disclosure happens before dispatch and names every exact Step, prompt-rule Step, attachment, tool grant, and destination provider.
- Consent transfer is destination-specific and packet-specific. A candidate approves the exact minimum event set. The reviewer verifies manifest equality and `historyComplete` for that selected chain.
- Ordinary Send does not silently broaden to full ancestry. Full-chain transfer is a separate consented operation.
- Private traces referenced by `q` are excluded unless independently selected and authorized.
- Restricted relay ACL means access control, not end-to-end encryption. Do not label it private/E2EE. Encryption is outside this plan.
- Replay surfaces must not expose hidden local paths, environment variables, secrets, auth material, or raw tool arguments without policy-based redaction.
- Timing and paste signals are descriptive evidence. CAPTCHA or behavioral models may flag patterns later, but Zine must not present them as proof of cheating or human authorship.
- Candidates receive explicit retention, deletion, destination, provider, and fallback-channel terms. A Zine or adapter failure cannot count against a candidate.

### Operation-record trust, visibility, and retention

`TaskOperation` events are signer assertions protected by their Nostr event ID/signature. They are not provider-signed. Their linkage to an artifact is verified by matching task root, exact target head, attempt ID, request/result hashes, and the artifact Step's `operation` summary.

Validation has two boundaries:

- **Writer/apply-time validation** has the prepared request bytes, local journal, current owner operation stream, provider result bytes, and target head. It MUST verify the request/result hashes, a unique un-forked attempt state of `succeeded/pending-review`, explicit acceptance, CAS, and attempt idempotency before signing an artifact Step.
- **Standalone reader validation** never assumes those local bytes or operation records are available. It first validates the TraceNode's own signature, schema, snapshot, relationships, and hashes. It then reports operation linkage as `verified` when the eligible signed TaskOperation is present and matches; `linked-record-unavailable` when the Step asserts an operation but the sidecar was not supplied; or `invalid` when a supplied sidecar/signature/state/hash contradicts the Step. Request material is independently `request-bytes-verified` only when exact canonical request bytes are included in the consent packet and hash correctly; otherwise it is `request-hash-asserted`.

`linked-record-unavailable` and `request-hash-asserted` are evidence limitations, not structural invalidity of the signed file Step. Only `invalid` suppresses operation-verification claims. A relay is not required to carry local TaskOperations, and absence must never be rewritten as proof that no attempt occurred.

| Field class | Signed | Local replay | Consent packet | Relay by default | Retention |
|-------------|--------|--------------|----------------|------------------|-----------|
| attempt IDs, states, timestamps, adapter/model/session labels | Yes | Yes | selectable | No | task lifetime |
| exact context/rule Step IDs, request/result hashes | Yes | Yes | selectable; required for high-fidelity attempt claim | No | task lifetime |
| human/provider messages returned to Zine | Yes at operation checkpoint or artifact Step | Yes | selectable | No | task lifetime or explicit deletion request policy |
| approval requests/decisions and normalized tool event summaries | Yes | Yes | selectable | No | task lifetime |
| raw streaming/protocol frames | No; hash may be signed | diagnostics only | Never | Never | bounded local diagnostic TTL |

| credentials, tokens, inherited environment, secret tool fields | Never | Never | Never | Never | not retained |

`visibility: packet-eligible` means “may be selected after consent,” not “automatically sent.” Signed events cannot be field-redacted without changing identity, so packet-eligible event schemas must exclude secrets at creation. A reduced disclosure is a new signed derived report citing the source operation; it is never presented as the unchanged original.

### Consent packet semantics

A transfer manifest is a signed `TaskOperation kind: consent` with:

- destination identity and relay/endpoint;
- ordered roots, each `{eventId, closure: exact|prev-to-genesis}`;
- an explicit ordered list of additional TaskOperation IDs;
- a deterministic event list with event ID, kind, and canonical-event-preimage byte length;
- an optional ordered `blobs` list, each `{blobId, kind:request-bytes, attemptId, mediaType, encoder, canonicalLength, sha256}`;
- explicit exclusions and inaccessible items;
- retention/deletion terms, named provider, expiry, and authorization nonce;
- `manifestHash = SHA-256(RFC8785(manifest payload excluding manifestHash))`; the enclosing TaskOperation signature binds that hash and payload to the sender/candidate.

Traversal is deliberately narrow. `exact` includes only the named event. `prev-to-genesis` follows only `e…prev` within that trace from the named root through genesis. It does not follow q, reply, fork, merge, extracted-from, operation, folder membership, or relay-hint edges. Any context, reply, coin, private q target, or operation record is included only as its own reviewed root/ID.

The `roots` array preserves user review order and is not the transfer order. To build the deterministic `events` array, compute every root closure, union/deduplicate by event ID, add included `prev` dependencies, then run stable Kahn topological order with ready nodes sorted by `(traceGenesisId, prevDepth, eventId)`. Parents therefore precede children even when roots overlap. Exact events with no included prev dependency use their own event ID as `traceGenesisId` and depth 0. After artifact events, append explicitly selected TaskOperations in stable topological order over included `op-prev`/`op-merge` dependencies with event-ID tie-breaking. A cycle, missing declared dependency, or ambiguous owner-operation fork makes manifest construction fail.

Canonical event bytes mean the NIP-01 event-ID preimage: UTF-8 of compact JSON `[0,pubkey,created_at,kind,tags,content]` with the event's exact tag and content values. The SHA-256 of those bytes MUST equal the event ID. Transport JSON object serialization, property order, and whitespace are irrelevant and are never hashed as packet identity.

Exact request bytes are local-only unless the user separately selects their attempt in the consent preview. Selection materializes an immutable packet blob whose `blobId` and `sha256` both equal SHA-256 of the exact adapter request bytes; its metadata must match that attempt's signed request hash/media type/encoder. Because those bytes may contain private context, blobs are excluded by default, listed verbatim by size/destination/provider in consent, and may not contain credentials or secret tool fields. Redaction produces a new derived blob and cannot earn `request-bytes-verified` for the original attempt.

`historyComplete=true` for a root means every declared prev link from the selected head to genesis is present, signatures/event IDs verify, ownership continuity passes existing rules, and no link is duplicated or cyclic. It says nothing about q closure or unobserved provider activity. Packet completeness is true only when every root with `prev-to-genesis` is complete and every exact/operation entry is present.

Event chunks are keyed by `manifestHash:event:<eventId>`; blob chunks by `manifestHash:blob:<blobId>:<offset>`, with fixed chunk size named in the manifest. Both resume through a receiver bitmap/ack set. The receiver reconstructs each NIP-01 canonical preimage, recomputes every event ID, canonical byte length, blob length/hash, manifest hash, and completeness before signing a `TaskOperation kind: transfer-ack` containing that same manifest hash. `request-bytes-verified` is available only for a complete matching request blob. Partial receipt is visibly incomplete and cannot produce a reviewer “complete history” checkpoint. Revocation stops future fetch/authorization when the transport permits it but cannot retract bytes already received; consent copy must say so. Republishing sends unchanged signed events. Any transformed/redacted derivative has new IDs and must cite its source packet.

## Concurrency and Consistency

- At most one directly mutating operation may be active per target file. Read-only operations and mutations of distinct targets may run concurrently.
- Prepared operations capture target head, target content hash, context Steps, prompt-rule Steps, action contract, provider configuration, and rendered request hash.
- Immediately before artifact apply, compare both target head and content hash. Mismatch produces `STALE_RESULT`; never auto-merge or last-writer-wins.
- Apply is atomic from the writer's perspective: the eligible `succeeded/pending-review` checkpoint, application decision, and resulting Step are linked transactionally or reconciled idempotently on restart.
- An attempt ID can map to at most one resulting artifact Step. A retry uses a new attempt ID and `retryOf`.
- Turn index assignment happens under the target mutation lock.
- New provider events arriving after cancellation are retained as late observable events but can never apply that cancelled attempt; using the content requires a new linked retry or an explicitly EXTERNAL manual incorporation.

## Performance and Storage

Unconditional cumulative snapshots are an intentional product choice. Optimize implementation without changing that semantic:

- content-address and deduplicate unchanged transcript chunks or blobs below the signed event layer;
- lazily materialize replay views and correspondence graphs;
- index relationship roles, task/turn IDs, and attempt IDs once at ingest;
- stream provider output into the attempt journal without repeatedly rewriting the artifact;
- create the cumulative artifact Step only on acceptance;
- keep compaction explicit and user-reviewed, not an automatic storage policy;
- benchmark a 1,000-turn task, a 10,000-event correspondence graph, mixed-role inbound queries, and a consent packet with a full selected ancestry.

Performance optimizations must not convert a complete signed snapshot into a reader-dependent partial turn representation.

## Design and UX Review

### Palette

Each model row shows provider/adapter, model, output pen, action, ordered prompt-rule chips, context defaults, permissions, and session state. Multiple rows can be visible. A user can define rows such as GLM Draft, Claude Critic, and Codex Research. The row's primary action remains one click; changing voice remains at most two clicks.

### Preflight and disclosure

Dirty references appear as a concrete list with three actions per item: open and Step, remove, or cancel. There is no “Step all automatically.” The final disclosure groups artifact context, prompt rules, attachments, tools, provider destination, and writable target.

### Replay

Replay distinguishes actor origin through both labels and voice/color, so meaning is not color-only. Context relationships are visible on the operation that used them and absent from later Steps unless explicitly selected again. External pastes use their assigned voice; internal quotations preserve source Runs.

### Correspondence

The default is still a readable file. A correspondence profile derives a letter/thread view, branch view, and citation graph from reply and q relationships. No new container semantics appear in the editor.

### Reviewer experience

The reviewer must explicitly complete the raw replay checkpoint before opening AI co-analysis. The analysis task and final report remain visibly separate from the candidate packet. The UI never uses “cheat,” “proof of human,” or reviewer-specific citation verbs.

### Empty and degraded states

- No model configured: explain local, GLM key, Codex, and Claude options without blocking manual writing.
- Adapter unavailable: keep the task readable and allow a different row/provider.
- Source receipt unresolved: offer EXTERNAL / UNVERIFIED paste or cancel; never relabel observed paste as manual input.
- Incomplete replay: show the exact missing/inaccessible events and suppress completeness claims.
- Unsupported historical local data after the clean cut: direct developers to reset/reimport the pre-production profile; do not invoke a hidden legacy reader.

## Dream State Delta

The current product has many of the right atoms—files, Steps, Runs, pens, model operations, q edges, replay, relay, MCP—but lacks a single coherent task/attempt model and precise relationship semantics. The largest delta is not raw model capability. It is binding existing prepared-context primitives to every live model action, making task history an ordinary cumulative file, and treating external harnesses as observable correspondents with bounded claims.

The dream state requires broad surface work, but only two fundamental new abstractions:

1. a durable typed operation record (task turn + attempt + prepared request); and
2. a declarative action/model-row registry over bounded contracts.

Everything else—Continue, Reply, Analyze, handoff, compaction, quotation, reviewer reports—is a disciplined use of those abstractions and the existing file graph.

## Temporal Interrogation Decisions

These hour labels describe when unresolved design questions would surface during an implementation session; they are not the delivery estimate. The delivery estimates and manual gates are below.

### Hour 1 — foundations (human ~1h / CC+gstack ~5–10min)

- The schema is a clean cut; implementation must not pause to design legacy aliases.
- Relationship roles are per q target, signed, and required for new mixed-role events.
- The cumulative task file is the canonical archive; the attempt journal is operational metadata, not a second chat truth.

### Hours 2–3 — core logic (human ~2h / CC+gstack ~10–20min)

- Context preflight blocks and never auto-Steps.
- Every dispatch receives an attempt ID; only accepted successful mutations Step.
- The target is revalidated immediately before atomic apply; stale output never auto-merges.
- Continue and Reply are distinct contracts available per turn.

### Hours 4–5 — integration (human ~2h / CC+gstack ~10–20min)

- Zine-hosted adapters are the high-fidelity path; MCP remains voluntary outside-in interoperability.
- Provider-native approvals are preserved and malformed flows deny by default.
- Executables are user-selected and identity-pinned; credentials remain provider-owned.
- Cross-provider switching uses a reviewed handoff manifest, not an invisible summary.

### Hour 6+ — polish/tests (human 2h+ / CC+gstack ~15–30min)

- Context-used edges do not become citation/social signals.
- Copy/paste preserves source Run styling without coining.
- Replay states evidence gaps and completeness honestly.
- Real-relay transfer, crash recovery, auth pilots, long-task performance, and accessibility are release criteria.
