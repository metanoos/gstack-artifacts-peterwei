## Relationship to the Focus Design

This is a follow-on to the approved singular-focus design. It preserves its
central rule:

> Focus chooses the subject. The player chooses a moment within that subject.
> Selection chooses directory items for an operation.

The MODEL lesson adds a second, independent question:

> Focus is where the model works. Mount is what it sees. Shield is what it must
> not see.

Focus never selects an item, and mounting never moves focus. In the directory,
the gold bar still means focus; the mount/inclusion glyphs describe prompt
context; neutral selection styling remains reserved for drag, right-click, and
batch operations.

## Problem Statement

Zine's first-run journey currently proves a model-free trace: edit, Step, and
replay. Its completion card offers **Add an AI model**, but that action dismisses
onboarding and drops the user into Models without explaining how a configured
provider relates to the MODEL voice, the focused document, Root mounting,
shielding, or Prompt Inspector.

The underlying controls already express a strong trust model, but the user must
infer it:

- one explicit Root/file/folder mount determines the context boundary;
- inherited descendants are included;
- shields recursively exclude private branches;
- Prompt Inspector renders the same message assembly used by a live operation;
- the MODEL voice records model-attributed additions in the trace.

That gap is especially costly for Zine's first commercial wedge: a writer doing
an AI-permitted audition must understand, before using AI, which draft is the
target and which supporting or private material crosses the provider boundary.

## Demand Evidence

Direct product demand remains unvalidated. No potential customer has completed
an uncoached Zine audition. This design is not evidence of demand; it removes a
comprehension and trust obstacle from the proposed validation workflow.

The interaction direction is supported by established onboarding guidance:

- Apple recommends short, optional onboarding that teaches through interaction
  and explains access at the moment it matters.
- VS Code distinguishes the active file from explicit file/folder context and
  describes context as everything available to the model.
- GitHub's content-exclusion guidance shows that users need explicit language
  about exclusions and their limitations, rather than a vague privacy promise.

Sources:

- https://developer.apple.com/design/human-interface-guidelines/onboarding
- https://code.visualstudio.com/docs/agents/concepts/context
- https://code.visualstudio.com/docs/chat/copilot-chat-context
- https://docs.github.com/en/copilot/concepts/context/content-exclusion

## Target User and Moment

The immediate user is a writer who has just completed Zine's model-free first
trace and elects to learn the AI workflow. For the commercial pilot, this is a
candidate beginning a paid, AI-permitted writing audition with a 24-hour
submission window.

The lesson succeeds when the writer can answer, before a workspace-bearing call:

1. Which trace will the model change?
2. Which files will be sent as context?
3. Which file is being withheld?
4. Which provider and operation will receive the request?
5. Where the model's contribution will appear in replay.

## Constraints

- Keep the existing three-step, model-free trace lesson intact and independently
  complete.
- Make the MODEL chapter optional, dismissible, and restartable from Help.
- Require deliberate gestures. Onboarding must not send workspace content merely
  because the user connected a provider or opened a coachmark.
- Reuse the live Models view, Press tree, context controls, Prompt Inspector,
  operation path, MODEL attribution, and trace player. Do not build a simulated
  onboarding-only copy of those systems.
- Use deterministic lesson material so every user can see one useful inclusion
  and one credible exclusion without risking their own private files.
- Do not change the protocol, wire format, key semantics, or trust semantics.
- Work in desktop and hosted web surfaces to the extent the existing provider
  and workspace features do.
- Preserve the singular focus contract and separate directory selection.

## Premises

1. The first trace should remain model-free so the user learns Zine's provenance
   primitive before adding provider complexity.
2. The completion card should offer a second optional chapter, not treat model
   configuration as completion of onboarding.
3. A provider connection test may send only a fixed synthetic probe such as
   `Reply with the single word: ok`; it must never include workspace content.
4. Prompt Inspector is the consent checkpoint before the first
   **workspace-bearing** provider call.
5. The lesson advances on verified outcomes, not on clicks or a generic Next
   button.
6. Focus, mount, shield, inspect, invoke, and replay are taught as one causal
   chain.

## Approaches Considered

### Approach A: Coachmarks over the user's current Root

Continue from Models into the existing workspace and point at whichever files
the user already has.

- Effort: small
- Risk: high
- Completeness: 5/10
- Advantage: no tutorial files.
- Rejected because the workspace may lack a useful source or a safe exclusion,
  and onboarding could expose real private content while trying to teach safety.

### Approach B: Deterministic context-boundary lesson

Create a small lesson folder, connect and test one real provider, focus a draft,
replace the broad Root mount with the lesson folder, shield a private note,
inspect and approve one immutable assembled request, invoke that exact request,
and replay the resulting MODEL contribution.

- Effort: medium
- Risk: low-to-medium
- Completeness: 10/10
- Chosen because it teaches the real mechanism with safe, predictable material
  and produces a genuine provenance result.

### Approach C: Inspector-only explanation

Open Prompt Inspector with explanatory annotations but do not require the user
to operate mount or shield controls.

- Effort: small
- Risk: medium
- Completeness: 7/10
- Deferred as a future quick tour for experienced users. It explains the result
  without building the motor and conceptual memory of shaping the boundary.

## Recommended Experience

![Approved rough interaction sketch](/tmp/gstack-sketch-zine-context-onboarding.png)

### Chapter entry

After the existing replay completes, keep **You made a trace** as the primary
achievement. Beneath it, offer:

- **Continue with a MODEL** — start the optional chapter;
- **Bring in a real file** — preserve the existing platform-dependent action;
- **Done for now** — dismiss without implying unfinished setup.

The description is concrete: “Connect a model, choose what it can see, and
replay its contribution.” Do not say “Add an AI model” without describing the
next outcome.

Choosing the MODEL chapter keeps onboarding active and navigates to Models.
It must not call `dismissOnboarding()`.

### Three visible phases, outcome-driven internal stages

Keep the visible journey at three phases while using finer internal states:

```ts
type ModelLessonStage =
  | "model-setup"          // MODEL 1 OF 3
  | "context-focus"        // MODEL 2 OF 3
  | "context-mount"        // MODEL 2 OF 3
  | "context-shield"       // MODEL 2 OF 3
  | "context-inspect"      // MODEL 3 OF 3
  | "context-run"          // MODEL 3 OF 3
  | "context-replay"       // MODEL 3 OF 3
  | "model-complete";
```

These extend `OnboardingStage`; they do not introduce another competing tour
state machine. Each transition is guarded by actual application state.

### Phase 1: Connect one model

Render a coachmark inside the real Models view:

- title: **Connect one model**;
- body: “Choose a provider and test it. The test sends only a fixed `ok` probe,
  never your Root.”;
- progress: **MODEL 1 OF 3**.

The user may add a preset or custom provider. A successful real connection test
calls an App-owned `onProviderTested(result)` callback. The result carries the
provider id, model id, protocol, endpoint/config fingerprint, and a per-test
nonce. The fingerprint covers every request-affecting provider field, including
an in-memory digest of the credential, but never exposes or persists the raw
credential. Test success is session evidence only; do not persist a permanent
“verified” claim because credentials and endpoints can become stale.

Only after a successful test in the current lesson session does **Use this model
in Press** become the primary action. That action:

1. assigns the provider to the MODEL voice through the same provider-selection
   path used by the Press;
2. creates or reuses the deterministic lesson folder;
3. navigates back to Press;
4. advances to `context-focus`.

If configured providers already exist, the user still tests the chosen provider
for this session. This adds one deliberate click but makes the rest of the
lesson deterministic and prevents onboarding from failing later with a stale
credential.

### Deterministic lesson material

Create an ordinary folder named `model-context-lesson`, adding `-2`, `-3`, and
so on if the name exists. It contains:

- `draft.md` — a short candidate launch brief with a sentence to extend;
- `source.md` — a useful public fact or house-style constraint;
- `private-note.md` — obviously private fictional material that the task says
  must not be sent.

Use the same honest bootstrap pattern as `onboarding-demo.ts`: the AUTHOR key
owns the traces, while system-provided starter prose is attributed to a neutral
starter voice, not AUTHOR or MODEL. Do not silently overwrite or delete prior
lesson traces. At completion, offer an explicit **Delete lesson files** action;
otherwise they remain ordinary inspectable traces.

The lesson content must be self-contained, fictional, and contain no network
fetched text. It must remain useful even if the provider has no current-world
knowledge.

### Phase 2: Focus and shape context in Root

Open the lesson folder in Root, but do not pre-complete the gestures being
taught. The user must click `model-context-lesson/draft.md` through the same
directory/tab transition defined by the singular-focus design. That click shows
the gold directory bar and gold tab top on the draft. It does not select it.

The ordinary broad Root mount may be present when the lesson begins. The next
gesture is to click the actual context control for `model-context-lesson/`,
replacing the broad mount with that folder as the one explicit mount. A call is
never allowed while Root or any unrelated branch remains effectively included.
The effective context must equal this exact allowlist before inspection:

```text
model-context-lesson/draft.md
model-context-lesson/source.md
model-context-lesson/private-note.md
```

After the private note is shielded, the effective sent-file allowlist becomes
exactly `draft.md` plus `source.md`. Sibling files elsewhere in Root are sentinel
exclusions and must never enter the canonical context block or its delta log.

The directory must distinguish the independent signals:

| Meaning | Visual | Behavior |
| --- | --- | --- |
| Focused target | gold bar at row left; gold top on exact tab | routes the operation and player |
| Mounted boundary | gold mount glyph | defines the one context root |
| Inherited inclusion | lighter gold context glyph | enters the context below the mount |
| Shield | purple shield/diamond glyph | recursively excludes that branch |
| Directory selection | neutral fill/outline | drag, right-click, and batch operations only |

The first coachmark points to the focused draft and states: **Focus is where the
model works.** It advances only when `draft.md` is the singular UI focus.

The next coachmark points to the lesson folder's actual context control and
states: **Mount what it sees.** It advances only when the explicit mount is the
lesson folder and the exact three-file pre-shield allowlist is effective. Status
copy and accessible labels name the mounted folder; color is supplemental.

The final Phase 2 coachmark points to `private-note.md`'s actual context control
and states: **Shield what it must not see.** It advances only when:

- the lesson folder remains the exact mount;
- `source.md` is effectively included;
- `private-note.md` is an effective shield boundary;
- `draft.md` remains focused.

Clicking a mount/shield glyph uses `stopPropagation`, changes context state, and
does not focus or select that row. If the user changes focus or mount state, the
coachmark explains the current mismatch and waits; it does not repair the state
silently.

### Phase 3: Inspect, approve, invoke, and replay

Point at the token estimate / inspect affordance and use the title **Inspect
what will be sent**. Opening the real Prompt Inspector for Extend must display:

- the selected provider and model;
- the focused target in the canonical context block;
- `source.md` as included context;
- no body, delta log, or metadata from `private-note.md`;
- the provider-card system layer, Zine operation layer, voice/lens layers, and
  user message assembled by the same helpers as the live operation;
- an honest note if a relay-backed portion of context could not be fetched.

The Inspector is a preview of Zine's assembled request. Copy must not claim to
control or reveal transformations performed later by the provider itself.

The Inspector's primary action is **Approve this request**. It creates an
immutable, session-only `PreparedOperation` after verifying the focused target,
trace head/content hash, exact effective allowlist, shield, provider
fingerprint, voice prompt, lens, operation inputs, canonical context block, and
fully prepared `messages[]`. Closing by Escape, backdrop, or the ordinary close
button does not approve or prepare anything.

After approval, **Extend** is the next real gesture and may send only that exact
`PreparedOperation`; it must not rebuild the prompt. Any change to focus, trace
head/content, mount, shields, provider configuration, MODEL voice, operation,
lens, or prompt layers invalidates the object, disables the prepared Extend,
and requires inspection again. The application still makes no workspace-bearing
call until the user deliberately clicks Extend.

Execution should use a lower-level `completePrepared(provider, messages)` path
that does not call `prepareChatMessages()` again. Prompt Inspector and transport
therefore consume the same immutable array instead of two coincidentally equal
rebuilds. Recording-transport tests compare the approved and transmitted
canonical JSON message arrays byte-for-byte.

At invocation, create a unique request id and capture the exact focused trace
head, content hash, provider fingerprint, and prepared-request hash. During the
tutorial request, temporarily lock edits to that lesson draft, focus retargeting,
the lesson mount/shield controls, and MODEL provider changes; keep **Cancel**
available. An external or relay-driven head change invalidates the result before
attachment. While the request is in flight:

- disable a second tutorial invocation;
- keep focus indicators stable;
- state that the shown context is being sent to the named provider;
- allow cancellation if the live operation supports it;
- do not advance on timeout, abort, empty response, or error.

On success against the captured head, the existing operation path appends
MODEL-attributed content and creates its normal provenance checkpoint. Record
the exact result node id plus a hash/range identity for the new MODEL span. Then
the coachmark points to the player: **Replay the contribution.** It advances
only when playback reaches that recorded result node and span—not merely any
MODEL-attributed frame. The player continues to follow focus; selection is
irrelevant.

The completion card says:

> You focused the work, shaped what the model could see, inspected the request,
> and recorded its contribution.

Actions: **Keep writing**, **Delete lesson files**, and **Replay again**.

## State and Navigation Rules

Keep sensitive/request material session-only while persisting only the minimum
safe resume identity:

```ts
type ModelLessonSession = {
  stage: ModelLessonStage;
  lessonFolderPath: string;
  testedProviderFingerprint?: string;
  testNonce?: string;
  prepared?: PreparedOperation;       // never persisted
  invocationId?: string;
  resultNodeId?: string;
  resultSpanHash?: string;
};

type ModelLessonResume = {
  version: 2;
  state: "available" | "interrupted" | "needs-replay";
  lessonFolderPath?: string;
  resultNodeId?: string;
  resultSpanHash?: string;
};
```

`PreparedOperation` contains the exact prepared messages and request-affecting
fingerprints but no raw secret. It is never serialized. The resume record may
identify an already-created lesson and completed result; it contains no API key,
messages, prompt body, provider response, or provider verification claim.

Guarded transitions:

| From | Verified event | To |
| --- | --- | --- |
| complete | user opts in | model-setup |
| model-setup | fixed-probe success + provider fingerprint | context-focus |
| context-focus | exact lesson draft becomes UI focus | context-mount |
| context-mount | exact lesson folder becomes sole mount/allowlist | context-shield |
| context-shield | exact private-note shield produces two-file sent allowlist | context-inspect |
| context-inspect | immutable request approved | context-run |
| context-run | matching request succeeds on captured head; exact result ids recorded | context-replay |
| context-replay | player reaches recorded result node and MODEL span | model-complete |

Every guard is re-evaluated before its side effect. Moving backward is allowed
when a dependency changes; no transition is based only on a DOM click.

- The existing trace lesson is complete before the MODEL chapter starts. The
  user may leave the optional chapter without losing that achievement.
- Within the same app session, navigation between Models, Press, Keys, and the
  Inspector preserves the current MODEL lesson stage.
- Closing the app during the optional chapter does not reopen an API-key or
  context coachmark automatically. On next launch, ordinary Press opens and
  Help offers **Resume model lesson** from a safe checkpoint.
- Resume never stores an API key in onboarding state. Provider storage continues
  to use the existing provider mechanism.
- Because connection verification is session-only, resume returns to
  `model-setup` and asks the user to test again unless `needs-replay` names an
  already-verified result node/span, which can resume directly at replay without
  making another model call. Existing lesson files may be reused after
  validating their exact manifest, paths, and content hashes; otherwise create
  a numbered folder.
- Dismiss is available at every coachmark and returns to ordinary product UI.
- Help exposes two explicit entries: **Replay first trace guide** and **Learn
  MODEL context**. Restarting either journey creates numbered tutorial material
  and never overwrites user work.

## Trust and Privacy Contract

The lesson must make these statements true and testable:

1. Provider configuration alone sends no Root content.
2. The connection test sends a fixed synthetic probe only.
3. A workspace-bearing call occurs only after a deliberate operation gesture.
4. The user approves the named provider and exact immutable Zine-assembled
   messages before that gesture.
5. Shielded content is absent from the canonical context block, file bodies,
   and aggregated delta log.
6. The operation transmits the approved `PreparedOperation` without rebuilding
   it. Dependency changes before invocation invalidate approval; conflicting
   tutorial controls are locked during invocation.
7. Errors are explicit. A partial context gather must never be presented as a
   complete preview.
8. Effective scope is the lesson folder only; unrelated Root siblings are absent
   from bodies, metadata, and the aggregated delta log.

The current `gatherContextBlock()` behavior skips a file's delta log after a
fetch failure. That is acceptable for ordinary best-effort operation only if it
is surfaced. For the onboarding consent checkpoint, expected lesson inputs must
be complete; a failure blocks the tutorial call with **Context incomplete — try
again**, rather than silently teaching that the preview was exhaustive.

## Architecture and Data Flow

```text
Trace complete
  -> user opts into MODEL chapter
  -> ModelsView tests fixed synthetic probe
  -> App receives tested provider/model/config fingerprint for this session
  -> existing MODEL provider assignment is updated
  -> lesson traces are bootstrapped through existing local/provenance path
  -> user focuses draft.md, mounts only the lesson folder, shields private note
  -> gather + live builders create one immutable PreparedOperation
  -> Prompt Inspector renders and approves that exact object
  -> user deliberately invokes Extend with the approved object
  -> matching result enters existing MODEL attribution + Step path
  -> exact result node/span identity drives focused replay completion
```

Implementation boundaries:

- `Onboarding.tsx`: phase-specific coachmarks and completion copy.
- `onboarding-state.ts`: extended stages, safe loading, and restart/resume intent.
- `ModelsView.tsx`: successful-test callback and gated **Use this model in
  Press** action; no onboarding-specific provider implementation.
- `App.tsx`: orchestration, provider assignment, lesson bootstrap, focus/context
  guards, Inspector observation, operation success, and replay success.
- `op-prompts.ts` / `llm.ts`: a shared preparation result and non-mutating
  `completePrepared` execution boundary so inspected and transmitted messages
  are the same object/value, not separately assembled equivalents.
- a new `model-context-lesson.ts`: pure names/content, collision-safe creation,
  and expected-state helpers, following `onboarding-demo.ts` conventions.
- existing `scope-model.ts`, `context-gather.ts`, `PromptInspectorModal.tsx`,
  operation builders, provider store, voice-provider store, and replay path
  remain authoritative.

No protocol package or wire-format change is required.

## Edge Cases and Failure States

- **Provider test fails:** remain in Phase 1, show the provider error next to
  the card, and allow edit/retry or another provider.
- **Provider returns text other than `ok`:** transport success counts if the
  existing test semantics count it; show the reply without claiming semantic
  verification.
- **Provider is removed after test:** return to Phase 1.
- **User changes provider in Press:** pause and ask them either to use the tested
  provider or explicitly retest the new one.
- **Provider/model/config changes after inspection:** discard the prepared
  operation and require a fresh provider test and inspection as appropriate.
- **Lesson path collision:** create a numbered folder; never merge into or
  overwrite an existing folder.
- **Focus changes before invocation:** invalidate any prepared operation; player
  and ordinary operations follow the new focus while onboarding explains how to
  refocus `draft.md`.
- **Directory items are selected:** leave selection intact unless an ordinary
  focus click clears it under the focus design; never treat it as lesson progress.
- **Mount is cleared or replaced:** invalidate any prepared operation, explain
  the effective state, and wait for the lesson-folder mount gesture.
- **Private note is focused:** focus changes, shield state does not; no call may
  proceed until the draft is refocused.
- **Context gather is partial:** block the tutorial operation and retry; never
  label a partial preview exhaustive.
- **Model call fails/aborts/returns empty:** preserve the pre-call trace and
  remain at `context-run`.
- **User attempts to edit during an in-flight tutorial call:** the lesson draft
  and request-affecting controls are temporarily locked; offer Cancel. If an
  external/relay head change still occurs, reject automatic attachment and
  return to inspect with an explicit conflict message.
- **Replay loading fails:** retain focus and the MODEL result, show retry, and do
  not mark the chapter complete.
- **Prior lesson material exists:** offer reuse only if an exact lesson manifest
  matches; otherwise create a numbered copy.
- **No provider feature on a surface:** keep the first trace lesson complete and
  explain that the optional MODEL chapter is unavailable there.

## Verification Plan

### Pure state and unit coverage

- Every new stage serializes, validates, dismisses, and restarts correctly.
- Interrupted model onboarding does not auto-open sensitive setup on launch.
- Existing providers still require a successful current-session test.
- Test results include provider/model/config fingerprint and nonce; failure,
  configuration mutation, and provider removal cannot advance the lesson.
- Lesson folder naming is collision-safe and never overwrites prior work.
- Starter prose attribution remains distinct from AUTHOR and MODEL.
- Focus, mount, inherited inclusion, shield, and directory selection remain
  independent.
- Mount/shield gestures do not activate or select directory rows.
- Before shielding, the exact effective allowlist is the three lesson files;
  after shielding, it is exactly `draft.md` plus `source.md`.
- Sentinel private files beside and above the lesson folder are absent from
  bodies, metadata, and every aggregated delta-log entry.
- Partial context gather blocks the tutorial call.
- No workspace call occurs before an explicit operation gesture.
- The prepared request is invalidated by every request-affecting dependency:
  focus, head/content, mount, shield, provider/model/config, voice, lens, op, or
  context-gather completeness.
- A recording transport receives the approved canonical JSON `messages[]`
  byte-for-byte; execution does not add, remove, reorder, or rebuild a layer.
- Operation failure, abort, or empty result does not advance.
- Concurrent local tutorial edits are locked; an external head change rejects
  attachment and requires reinspection.
- Only the recorded result node id plus MODEL span hash advances to replay.
- Only replay reaching that exact node/span completes the chapter.

### UI integration coverage

- Completion shows the optional MODEL chapter without demoting the first trace
  achievement.
- Choosing it routes to Models without dismissing onboarding.
- The fixed probe contains no workspace context.
- Successful test exposes **Use this model in Press** and assigns the correct
  MODEL provider.
- The gold focus bar/tab and context glyphs match their distinct meanings.
- The learner performs focus, lesson-folder mount, and private-note shield using
  the real controls. Each state has text/ARIA status and keyboard operation;
  color alone is never the instruction.
- Prompt Inspector names provider/model and faithfully shows expected message
  layers and canonical context.
- `private-note.md` and unrelated Root sentinel content/log data are absent from
  the preview and recording transport request.
- Ordinary close does not approve or invoke; **Approve this request** prepares;
  clicking Extend invokes that prepared request exactly once.
- The resulting span is MODEL-attributed and replayable.
- Dismiss, Help restart, resume, and explicit lesson deletion all work.

Update the existing onboarding tests that currently assert there is no context
lesson. Run client unit tests first, then the client build. A real-relay smoke is
required for provenance attachment/replay, and a recording-transport integration
test is required for the privacy boundary. Exercise one real provider before
release; if credentials are unavailable in CI, record that manual gate rather
than weakening it to “when feasible.”

## Success Criteria

### Interaction correctness

- No workspace content reaches a provider before a deliberate operation click.
- Every tutorial call uses the provider, focus, and effective context shown at
  its consent checkpoint, from the same immutable prepared request.
- A shielded lesson note and unrelated Root siblings are absent from both
  preview and actual request.
- The player replays the exact recorded MODEL-attributed addition from the
  focused draft.
- The first trace course remains useful and complete for users who decline AI.

### Product validation

Observe five new writers without coaching. Before they click Extend, ask each to
predict the target trace, included source, excluded note, and receiving provider.

The design passes the first usability gate when:

- four of five answer all four correctly;
- all five understand that Test sent a synthetic probe but Extend sends the
  displayed workspace context;
- at least three can later reproduce focus/mount/shield behavior in a different
  folder without the coachmark;
- no participant believes directory selection controls the model target;
- at least two participants ask to use the workflow on a real audition or
  writing task.

These observations validate comprehension. They do not yet validate commercial
demand for the audition product.

## Distribution Plan

Ship the optional chapter only on surfaces where a provider can be configured
and the same Prompt Inspector/live operation path is available. Do not add a
separate hosted tutorial backend. Use the existing app release path and keep the
model-free first trace as the universal fallback.

For the first commercial pilot, onboarding happens before the 24-hour audition
clock begins. Setup time and provider troubleshooting must not consume the
candidate's paid submission window.

## Open Questions

- Whether tutorial cleanup should delete immediately or move traces through the
  existing Oblivion flow; follow the product's ordinary deletion semantics.
- Whether provider connection tests should eventually have a non-LLM health
  check. This design uses the live completion path because it proves the
  configured protocol actually works.
- Whether the read-only reviewer press should expose the context boundary used
  for each historical MODEL operation. That is valuable for the audition wedge
  but is separate from first-run author onboarding.

## The Assignment

After implementation, put a new writer in front of a fresh press. Let them make
the first trace and opt into the MODEL chapter without coaching. Immediately
before Extend, ask: “What exactly is about to leave this press, where is it
going, and which trace will change?” Record their answer before allowing the
call. After replay, give them a new folder and ask them to focus one draft,
include one source, and shield one note without prompts.

## What I noticed about how you think

- You did not want model setup to end at a credentials form; you wanted the
  person to understand the causal system that makes the model trustworthy.
- You treated Root mounting and shielding as teachable product primitives, not
  advanced settings.
- You kept returning to the same unifying idea: author gestures, the player,
  model operations, and onboarding should all center on focus.
- You accepted a deterministic lesson because privacy comprehension matters
  more here than making onboarding look magically personalized.
