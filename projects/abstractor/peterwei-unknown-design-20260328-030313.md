## Problem Statement

`abstractor` already has two useful ingredients:

- a toy resonance/decoherence model for a persistent self vector
- a terminal-first fork-lab surface that lets multiple branches diverge from the same prompt

What it does not yet have is a research harness that treats the self as a finite, lossy abstraction of experience and then selects for better abstraction over time.

The proposed next step is a three-timescale system:

- an **outer loop** that evolves projection genomes and update hyperparameters by debate fitness
- an **inner loop** that runs a live branchable inference process where a self state `S` conditions what gets attended to
- an **S-update loop** between inferences where `S` resonates with recurring structure and decoheres from noise or contradiction

The point is not to "solve consciousness." The point is to build a legible, runnable system where persistent identity changes what the model notices, says, and preserves.

## What Makes This Cool

The "whoa" moment is not just that branches produce different text.

The "whoa" moment is:

- same prompt
- same base harness
- different self orientation
- visibly different attention emphasis and text continuation
- a saved artifact showing why the divergence happened
- an outer loop that keeps the genomes whose selves compress experience into more useful abstractions

In your framing, `S` is not memory as storage. It is memory as **lossy compression under fixed representational capacity**. Good selves keep deep structure. Bad selves keep trivia.

## Constraints

- Stay terminal-first for the first real version. The current repo already has the right product shell in `self-fork-lab`.
- Reuse the current Python package structure and additive `consciousness_loop` layer rather than introducing a second parallel system.
- Keep low-level vector helpers small and local in `abstractor/model.py`.
- Do not require a from-scratch training stack or full transformer implementation in v1.
- Produce typed JSON artifacts that can be replayed, inspected, and scored later.
- Keep the demo comparative and causal. A user must be able to point at a branch change and explain why it happened.
- Treat "smart" as measurable proxies, not vibes.

## Premises

1. The first deliverable should be a research harness inside the existing fork-lab and self-vector surfaces, not a fresh training framework.
2. The first thing worth proving is that a persistent `S` measurably changes branch behavior beyond ordinary prompt memory baselines.
3. The outer loop should optimize observable proxies for abstraction quality, not philosophical claims.
4. The product surface should stay comparative and legible: same prompt, different `S`, visibly different future.

## Approaches Considered

### Approach A: Fork-Lab Research Harness

Keep the current fork-lab product surface and add a toy three-timescale engine behind it.

- **Effort:** M
- **Risk:** Low
- **Pros:** Reuses the most code, ships a showable demo fast, preserves the branch artifact workflow already emerging in the repo
- **Cons:** The attention story may be approximate in early versions, and skeptics may say it is still an external controller unless the causal evidence is explicit
- **Reuses:** `consciousness_loop/self_fork_lab.py`, `consciousness_loop/self_vec_experiment.py`, branch runtime state, artifact schema, Rich renderer, vector update helpers

### Approach B: Self-Conditioned Attention Core

Build a new toy attention engine where `[S | x_1 ... x_n]` is processed directly with explicit self-conditioned projections and visible attention-state effects.

- **Effort:** L
- **Risk:** Med
- **Pros:** Best test of the thesis that identity changes perception itself, strongest story for mechanistic interpretability readers
- **Cons:** More math surface area, more net-new code, harder to keep small and verifiable
- **Reuses:** UI shell and artifact ideas, but less of the current live MLX path

### Approach C: Identity Controller Over Existing LLM

Keep the MLX/Qwen backend and let `S` bias retrieval, token ranking, and branch steering without claiming to be inside attention proper.

- **Effort:** M
- **Risk:** Med
- **Pros:** Easiest way to get interesting text, practical for demos, easiest bridge from current code
- **Cons:** The thesis gets weaker because identity becomes steering instead of perception
- **Reuses:** existing MLX backend path, live token loop, branch renderer, JSON artifacts

## Recommended Approach

Choose **Approach A**.

It is the smallest version that still preserves the real idea. The repo already has the correct shell: branchable live runs, self snapshots, artifacts, and a terminal UX that makes comparative behavior legible. The right move is to deepen that shell into a research harness rather than throwing it away for a full architecture rewrite.

Approach A should absorb one key lesson from Approach B: the system must expose at least one causal trace showing that self orientation changed what the branch focused on or preserved. Otherwise the whole thing collapses into "prompt memory with philosophy attached."

The first scoring loop should treat "smart" as a composite fitness made of:

- stronger-model pairwise debate preference
- divergence quality, branches should not collapse into the same answer
- self persistence, the branch should retain a coherent orientation over multiple runs
- economy of expression, shorter clearer abstractions beat bloated surface-detail dumps

The genome should include both projections and self-update behavior:

- `W_q`, `W_k`, `W_v`
- `tau_resonate`
- `tau_decohere`
- resonance and decoherence strengths
- normalization constant
- `dim(S)`

That keeps the evolutionary loop honest. It is not just tuning prompts. It is selecting for better compression under fixed self capacity.

## Open Questions

- How "inside attention" does the first version need to be to satisfy the thesis without forcing a full transformer rewrite?
- Should the first branch intervention mutate `S`, mutate the genome, or inject a cue that perturbs `S` indirectly?
- How should branch collapse be penalized in the fitness score so the system does not learn to game the judge with stylistic sameness?
- Should the first stronger-model judge live inside the harness, or should the first version use deterministic local heuristics plus optional judge integration?
- What is the cleanest baseline to compare against: plain branch seeds, prompt-only steering, or static memory without resonance/decoherence?

## Success Criteria

- `self-fork-lab` can run a three-timescale demo without changing the repo's basic package structure.
- Each run writes an artifact that includes prompt, branch texts, self snapshots, interventions, genome parameters, and scalar fitness components.
- A user can trigger or replay at least one intervention and see branch behavior change in the next visible steps.
- The outer loop can rank genomes with a composite score made from judge preference plus compression-quality metrics.
- The system can demonstrate at least one case where branches differ for a reason more specific than random seed alone.
- Unit tests cover genome serialization, score composition, `S` update behavior, and artifact shape.

## Distribution Plan

The first deliverable remains a Python CLI, centered on the existing `self-fork-lab` console entry point defined in `pyproject.toml`.

Users get it through the current editable install path:

- `pip install -e .`
- `pip install -e '.[autoresearch]'` when MLX-backed paths are required

Artifacts remain local JSON files under `consciousness_loop/fork_runs/`, which makes them easy to inspect, diff, and eventually replay.

CI/CD already exists:

- `.github/workflows/ci.yml` runs `python -m unittest discover -s tests -v` and `python -m build`
- `.github/workflows/publish.yml` builds and publishes tagged releases to PyPI

That means the distribution story is already there. The design should extend the existing CLI surface and test/build pipeline instead of inventing a second one.
