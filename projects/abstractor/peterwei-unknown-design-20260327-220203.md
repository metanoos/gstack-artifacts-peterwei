## Problem Statement

`abstractor` currently contains the ingredients for a persistent-self simulation, but not yet a product-level experience. The goal is to turn it into a terminal-first interactive playground for ML hackers, where one prompt can be run through multiple persistent selves and the user can watch those trajectories diverge token by token.

This is not a generic chat UI and not a pure research harness. The core experience is comparative: same prompt, different self state, visibly different path.

## What Makes This Cool

The "whoa" moment is branching. A user runs one prompt through several self initializations or parameter presets and watches the generated language, similarity scores, and resonance/decoherence behavior split in real time.

That creates a concrete artifact people can inspect, screen-record, and share:

- one prompt
- multiple selves
- visibly different trajectories
- a legible explanation for why they diverged

## Constraints

- The first version should stay terminal-native. No web app first.
- The existing Python package structure should be reused rather than replaced.
- Core vector math already lives in `abstractor/model.py` and should stay there.
- Live token-driven orchestration already lives in `consciousness_loop/self_vec_experiment.py`, which is the natural extension point.
- Optional MLX-backed generation exists, so the design should work with that path first.
- The audience is technical ML/AI hackers, so polish matters less than legibility and immediacy.
- Distribution must eventually become a real CLI surface, not just a one-off dev script.

## Premises

1. The first version should be a terminal-native interactive CLI, not a notebook and not a web app.
2. The core artifact is the evolving self trajectory, not just generated text or benchmark scores.
3. The killer interaction is branching the same prompt through different initial selves or parameter settings and watching the traces diverge.
4. The existing seams already support this: `abstractor/model.py` owns the vector math, and `consciousness_loop/self_vec_experiment.py` is the right place for live token-driven orchestration and trace capture.
5. Distribution should be a real CLI people can install and run, not just a dev-only script. Packaging and a sane command surface matter from day one.

## Approaches Considered

### Approach A: Live Trace CLI

Extend the current single-run experiment into a polished terminal command that streams token-by-token updates, resonance/decoherence state, similarity, and summary stats in real time.

- Effort: Medium
- Risk: Low
- Pros:
  - Smallest diff, fastest path to something people can run this week
  - Reuses the current `SelfVecConfig`, `SelfVecStep`, `SelfVecResult`, and `run_self_vec_experiment()` path almost directly
  - Easy to test with the existing fake backend pattern
- Cons:
  - Less differentiated, could feel like debug output with better formatting
  - Branching becomes an add-on instead of the main event
  - Harder to evolve into multiple views without later refactoring
- Reuses:
  - `consciousness_loop/self_vec_experiment.py`
  - existing argparse CLI pattern
  - existing result dataclasses

### Approach B: Trace Engine + Renderer Split

Separate the live experiment into a reusable trace engine that emits full branchable state, then add a terminal renderer on top. This is the clean architecture path.

- Effort: Large
- Risk: Medium
- Pros:
  - Best long-term shape if later versions add notebook, web, or artifact export views
  - Makes branching, replay, and saved traces first-class instead of hacked on
  - Keeps model logic, orchestration, and presentation from collapsing into one file
- Cons:
  - More up-front design work
  - Slower to first "whoa"
  - Easy to overbuild if the interaction turns out not to be compelling
- Reuses:
  - `abstractor/model.py` update math
  - `SelfVecBackend`
  - current CLI config/result objects as seeds for a cleaner API

### Approach C: Self Fork Lab

Make branching the product from day one. Same prompt, multiple initial selves or parameter presets, shown side by side in the terminal as they diverge token by token with live similarity and resonance traces.

- Effort: Medium to Large
- Risk: Medium
- Pros:
  - Most distinctive and most aligned with the intended "whoa" moment
  - Turns the evolving self into something visible and comparative
  - Strong demo energy for ML hackers, because the same prompt producing different trajectories is immediately legible
- Cons:
  - Needs tighter UX design to avoid terminal chaos
  - More moving parts than a single-run CLI
  - Requires discipline to keep the first version small
- Reuses:
  - current one-token loop in `consciousness_loop/self_vec_experiment.py`
  - seed-driven self initialization
  - projection/update path
  - test fake backend pattern

## Recommended Approach

Choose Approach C, Self Fork Lab.

This is the actual wedge. A polished single-run trace is fine, but it is not memorable. The branching experience is what makes the repo feel alive and makes the concept legible in one glance. If the same prompt stays stable in one branch, turns reflective in another, and decoheres into a stranger path in a third, the product immediately explains itself.

The recommended v1 shape:

- New top-level CLI mode focused on branching, for example `python -m consciousness_loop.self_fork_lab`
- Accept one prompt plus three fixed seed-based branches in v1
- Render three synchronized branch panes in the terminal
- For each branch, show generated continuation, average similarity, resonance/decoherence balance, and latest token
- Maintain an event log for exact numeric inspection
- Save a replayable JSON trace artifact after each run

### V1 Decisions Locked During Review

- Branch count is fixed at 3 in v1 for readability.
- Branches vary by seed only in v1. Threshold and strength stay shared.
- The runtime is MLX-only for real runs in v1, with explicit CLI messaging when MLX is unavailable.
- The package exposes a real console-script entry point in addition to `python -m ...`.
- The fork lab lives in a new `consciousness_loop/self_fork_lab.py` module, while shared branch advancement stays in `consciousness_loop/self_vec_experiment.py`.
- The live renderer uses Rich, but rendering stays in the CLI layer rather than leaking into the engine layer.
- Each run writes one typed JSON artifact. Replay support is deferred until after the initial live flow ships.

## Information Architecture

The product has two distinct moments. They should not compete for the same pixels.

1. Preflight: define the experiment clearly and give the user confidence to start it.
2. Live run: maximize legibility of divergence and reduce everything else to supporting context.

### Screen Structure

```text
SELF FORK LAB
=============

[Screen 1: Preflight]
  ├── Prompt input
  ├── Branch summary (A/B/C seeds only in v1)
  ├── Run controls
  │    ├── model
  │    ├── max tokens
  │    └── artifact path
  ├── "What to watch for" hint
  └── Primary action: Start run

                |
                v

[Screen 2: Live Run]
  ├── Run header
  │    ├── prompt excerpt
  │    ├── model
  │    ├── token budget / progress
  │    └── artifact destination
  ├── Primary region: 3 branch panes
  │    ├── branch A
  │    ├── branch B
  │    └── branch C
  ├── Secondary region: event log
  └── Tertiary region: run summary / save confirmation

                |
                v

[Screen 3: Run Complete]
  ├── Final branch summaries
  ├── Saved artifact path
  ├── Suggested next action: replay or inspect JSON
  └── Suggested next action: rerun with a new prompt
```

### Live Run Hierarchy

The live run screen must make the branch panes the unmistakable hero.

1. First thing the user sees: branch text divergence.
2. Second thing the user sees: per-branch summary metrics that explain why the branches are separating.
3. Third thing the user sees: exact numeric event log for inspection.

Controls are not part of the live hierarchy. Once the run starts, setup collapses into a one-line run header so the branches own the width.

The event log is inspection tooling, not the hero. During the live run, the default is a compact one-line status strip that surfaces the latest branch action, current token step, and any warning or completion status. The fuller multi-row event log appears only when explicitly requested or once the run completes.

Each live branch pane shows a rolling recent-text window rather than the full accumulated branch transcript. The purpose of the pane is to keep the newest divergence visible. Full branch text belongs in the saved artifact and the completion view.

### Layout Budget

The v1 layout should treat terminal width as a budget, not an infinite canvas.

- Best case width: 3 branch panes side by side, event log below.
- Best case width: 3 branch panes side by side, with a single-line status strip below them.
- Medium width: 3 branch panes remain visible, but branch footers compress to a smaller summary vocabulary.
- Narrow width: degrade to one branch-focused stacked view with branch cycling or vertical ordering, while preserving status and latest divergence cues.

### Navigation Flow

```text
launch command
  -> preflight screen
  -> start run
  -> live run screen
  -> run complete screen
     -> save artifact confirmation
     -> rerun
     -> inspect later
```

## Interaction State Coverage

The terminal UI has to be legible when nothing is happening yet, when one branch finishes before the others, and when the runtime cannot start at all. The user should never have to infer state from missing text.

### State Table

```text

-------------------------|--------------------------------------------------|--------------------------------------------------|----------------------------------------------------|-----------------------------------------------------|--------------------------------------------------
Preflight screen         | n/a                                              | prompt input visible, 3 seed chips visible,      | inline validation message under the field that     | n/a                                                 | n/a
                         |                                                  | run button disabled until prompt is non-empty    | failed, run button remains visible                 |                                                     |
Model startup            | header says "loading model...", branch panes     | n/a                                              | full-width blocking message: "MLX runtime          | header flips to live token progress                 | n/a
                         | show placeholder text and pulse-free status copy |                                                  | unavailable on this machine" plus install action   |                                                     |
Live branch pane         | branch header says "starting", body shows        | branch created but has not emitted a token yet   | branch header says "error", body shows short       | branch header says "complete", footer locks final   | branch header says "complete early" while other
                         | waiting copy, footer reserves metric slots       | with "waiting for first token" copy              | failure reason, other branches continue            | metrics and latest cue token                        | branches continue running

```

### State Copy Rules

- Loading copy should describe what the system is doing in plain language, not just show a spinner surrogate.
- Empty or idle states should invite the next action, not just announce absence.
- Error states must name the problem and the next move, especially for MLX runtime failures and artifact write failures.
- Partial states must be explicit when one branch finishes or fails before the others, so the user never mistakes silence for a freeze.
- A single branch failure does not abort the whole run in v1. The failed branch stays visible in-place with an explicit failure label and short reason, while the remaining branches continue.

### State Hierarchy

When a run is live:

1. Branch text remains the hero, even in partial states.
2. Branch status is the first supporting signal.
3. The status strip carries global warnings, completion messages, and save results.

### First-Run Guidance

The preflight screen should include one short "what to watch for" hint and one short runtime expectation hint:

- "Same prompt, different self history. Watch where one branch stays coherent and another drifts."
- "MLX models can take a moment to warm up before the first token appears."

## User Journey & Emotional Arc

The product should feel like a controlled experiment, not a noisy dashboard. The user is not browsing. They are running one prompt to watch a split happen and to understand it fast.

### Storyboard

```text
STEP | USER DOES                     | USER FEELS                         | PLAN RESPONSE
-----|-------------------------------|------------------------------------|------------------------------------------------------------
1    | Launches the command          | Curious, slightly skeptical        | Preflight screen is calm, specific, and low-chrome
2    | Reads the prompt + branch set | Oriented                           | Fixed 3-branch setup makes the experiment legible instantly
3    | Starts the run                | Anticipation                       | Short startup copy explains warmup without panic
4    | Watches first tokens appear   | Relief, then curiosity             | Branch panes update in lockstep so divergence is easy to spot
5    | Sees one branch drift         | "Oh, I see it"                     | Branch text remains the hero, metrics explain the drift
6    | Sees one branch finish/fail   | Wants clarity, not mystery         | Explicit in-place branch state explains what happened
7    | Run completes                 | Wants takeaway, not just raw data  | Completion view summarizes each branch and confirms saved artifact
8    | Leaves the tool               | Feels this is replayable and worth sharing | Exact artifact path plus rerun / inspect-next actions
```

### Time-Horizon Design

- First 5 seconds: the CLI should read like one intentional experiment. Prompt, branch labels, and a single "what to watch for" line are enough.
- First 5 minutes: the user should be able to understand divergence from the branch panes alone, without needing the event log.
- Long-term relationship: saved artifacts and repeatable prompts should make the tool feel like a lab instrument worth coming back to, not a one-off demo.

### Emotional Design Rules

- The interface should feel calm before it feels impressive.
- The first visible success is not "tokens are streaming." It is "I can tell these branches are meaningfully different."
- Completion should reward the user with a concise explanation, not just a dead terminal and a file path.
- Any failure state should preserve trust by being plain, local, and recoverable.
- The completion screen leads with a plain-language branch summary, then shows the saved artifact path and next actions beneath it.

## Visual Direction & AI-Slop Guardrails

The terminal UI should feel like a lab instrument, not a generic admin dashboard. The composition has one job: make divergence legible fast.

### Classifier

This is **APP UI**, not a marketing page.

That means:

- calm surface hierarchy
- low chrome
- few colors
- typography and spacing do more work than borders
- cards and panels only exist when they clarify grouping

### Hard Rules

- Branch panes are the only large visual regions that deserve full borders during the live run.
- The run header and status strip should be visually lighter than the branches.
- The event log should not appear as a second hero region.
- Decorative borders, repeated micro-panels, and ornamental labels should be cut.
- If removing one border or one label improves readability, remove it.

### Litmus Checks

```text
1. Product unmistakable in first screen?        YES, if branch panes dominate.
2. One strong visual anchor present?            YES, the branch trio.
3. Understandable by scanning headings only?    Must be yes.
4. Each region has one job?                     Must be yes.
5. Are cards actually necessary?                Only for branch grouping.
6. Does motion improve hierarchy?               Only if it marks new tokens or status changes.
7. Would this still feel premium without lines? It should, because hierarchy comes from layout first.
```

### Terminal-Specific Slop Risks To Avoid

- Box around everything.
- Every label in all caps.
- Equal emphasis on controls, event log, and branch text.
- More than one accent color carrying meaning.
- Footer metrics multiplying into a miniature dashboard under each branch.
- Status copy that reads like internal debug output instead of guidance for a user.

### Typography and Color Rules

- Use monospace for the product surface because it fits the lab-tool feel and the repo context.
- Use weight, spacing, and sparse color accents to create hierarchy before adding more lines.
- Use color only as reinforcement. Status must still read correctly in plain monochrome terminals.
- Reserve the strongest visual accent for branch state changes and final success or failure notices.
- During the live run, each branch footer shows only two always-visible support signals:
  - latest cue token
  - one compact drift or resonance summary

Anything more detailed belongs in the completion screen or explicit inspection mode, not under every branch at all times.

### Motion Rules

- No constant pulse or spinner theatre.
- The only live motion should be meaningful: token appearance, branch status change, or save confirmation.
- If motion draws more attention than the divergence itself, it is wrong.

## Design System Alignment

There is no project-level `DESIGN.md` yet, so this plan needs a small terminal design vocabulary baked into it.

### V1 Terminal Vocabulary

- **Primary region:** the three branch panes
- **Secondary region:** compact status strip and completion summary
- **Tertiary region:** inspection details such as the event log or artifact path

### Text Style Roles

- **Title style:** used only once, for the screen title or completion summary heading
- **Label style:** short, low-emphasis labels for model, tokens, or branch identity
- **Body style:** branch text and plain-language summary copy
- **Status style:** success, warning, or error messaging with optional accent color

### Component Rules

- Branch panes are the only heavy container component during the live run.
- The status strip is a lightweight horizontal component, not a second panel stack.
- The completion summary is a textual summary block first and a metadata block second.
- The preflight screen may use grouped sections, but the live run should avoid repeated side panels.

### Consistency Rules

- The same branch naming scheme must be used everywhere: branch A, branch B, branch C.
- The same metric names must appear in live run, completion, artifact, and replay views.
- Copy should consistently describe divergence in human terms: stable, drifting, completed early, failed.

### Locked Copy Vocabulary

- Branch naming: `branch A`, `branch B`, `branch C`
- Mid-run branch states: `starting`, `running`, `completed early`, `completed`, `failed`
- Divergence descriptors: `stable`, `drifting`, `pulled away`, `held close`
- Artifact messaging: `trace saved`, `save failed`, `inspect later`
- Follow-up actions: `rerun`, `inspect JSON`, `replay later`

Avoid inventing synonymous labels in adjacent views. The product should read like one voice, not a stitched-together toolchain.

## Responsive & Accessibility

Terminal-native does not remove responsive and accessibility requirements. It changes their shape.

### Width Breakpoints

```text
WIDTH                    | LAYOUT BEHAVIOR
-------------------------|--------------------------------------------------------------
120+ columns             | 3 branch panes side by side, compact status strip below
90-119 columns           | 3 narrower branch panes, reduced footer detail, tighter labels
70-89 columns            | stacked branch view with one branch per row, status strip retained
under 70 columns         | single-branch focused view with explicit notice that the layout is condensed
```

### Responsive Rules

- The product never silently truncates the meaning of a branch state.
- When width is constrained, secondary details disappear before branch text does.
- The first thing sacrificed is inspection detail, then extra metric labels, never core branch identity or status.
- The narrowest layout must still show:
  - current branch name
  - current branch state
  - latest visible continuation text
  - one compact divergence summary
- In the narrowest layout, the UI shows one branch at a time in stable A -> B -> C order, with a plain-text notice that the layout is condensed and comparison continues in sequence rather than side by side.

### Keyboard and Interaction Rules

- The preflight screen must be fully operable with keyboard only.
- The primary action should be reachable without pointer assumptions.
- If the event log or replay mode expands later, its toggle and navigation need keyboard-accessible focus order.

### Accessibility Rules

- Color never carries meaning by itself. Status words must always be present in text.
- Minimum contrast target should be high enough for monochrome-ish terminals and washed-out themes.
- Avoid rapid flashing or constant motion in the live run.
- Any condensed-width notice should explain what changed, for example: "condensed layout, showing one branch at a time."
- Artifact paths and errors should be selectable and copyable as plain text.

### Screen Reader / Text Capture Considerations

- The completion summary should read coherently when copied as plain text.
- Branch order should stay stable so pasted logs remain understandable.
- Decorative separators should be sparse enough that copied output still makes sense in raw text form.

## What Already Exists

- The current sketch already establishes the right primary idea: branch divergence is the hero, not terminal decoration.
- Existing repo CLI surfaces are plain, textual, and low-chrome. The fork lab should extend that tone rather than invent a louder interface style.
- The eng review already locked several product-shaping decisions that this design should reuse:
  - fixed 3-branch v1
  - seed-only branching
  - explicit MLX-only runtime messaging
  - one typed JSON artifact per run
  - compact live status instead of a permanently expanded event log

## NOT in Scope

- A broader cross-project `DESIGN.md` for every future UI surface. Useful, but larger than this v1 feature.
- Rich replay mode design. Replay is deferred until after the initial live run experience ships.
- Decorative terminal chrome, animations, or non-essential motion. The product wins on legibility, not ornament.
- Per-branch preset editors or advanced setup workflows. V1 keeps branch setup intentionally narrow.

## Open Questions

- Should branch presets vary only the self seed in v1, or also vary threshold and strength parameters?
- Does v1 require a non-MLX fallback backend for contributors who cannot run the Apple-silicon path?
- What is the smallest useful saved artifact: raw JSON trace, plain-text replay, or both?
- Should branch count be fixed at 3 in v1 for readability, or configurable from the start?
- Is the first public distribution target local-dev users only, or should the design assume `pipx` / PyPI publication immediately after v1 works?

## Success Criteria

- A user can run one command locally and see at least three live branches for the same prompt.
- Divergence is legible without needing to inspect raw vectors.
- Each branch shows enough telemetry to explain what happened: token stream, similarity, resonance/decoherence state, and summary metrics.
- Every run emits a saved trace artifact that can be replayed or inspected later.
- The CLI remains understandable in one terminal window without requiring a notebook or web browser.
- The feature is covered by deterministic tests using fake backends rather than requiring a live model in CI.

## Distribution Plan

Initial delivery should be as an installable Python CLI within this package.

- Local/dev distribution:
  - `pip install -e .`
  - run with a stable module or console entry point
- Public distribution:
  - publish wheels to PyPI once the command surface settles
  - support `pipx install abstractor` as the easiest public install path

CI/CD is currently absent from this project. Public release automation should therefore be treated as a follow-on requirement:

- when the project is moved into a real git-hosted repository, add GitHub Actions for tests on push
- on version tags, build and publish the package to PyPI
- if public packaging is deferred, state that explicitly in the README so users know the first version is a local experimental CLI
