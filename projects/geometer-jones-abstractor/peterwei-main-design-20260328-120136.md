## Problem Statement
The repo already generates tournament history, promotion artifacts, and signal benchmarks for self-vector KV-bias experiments, but the evidence is split across multiple append-only files and partially overlapping payloads. That makes it too easy to tell a compelling story from scattered numbers without being able to replay exactly what happened, compare checkpoints cleanly, or inspect why a candidate "won" without rerunning the model.

The immediate goal is not to prove the full philosophical thesis. The immediate goal is to build a research-grade measurement layer that can answer a narrower question: does a persistent self-vector plus low-rank KV modulation produce reproducible behavioral and mechanistic gains over a frozen baseline, and can those gains survive replay and scrutiny later?

## What Makes This Cool
The cool version is not "more logs." The cool version is that a saved checkpoint becomes a portable scientific object.

One file should tell you:
- what self-vector and genome were tested
- what prompt bank and judge settings were used
- how the candidate behaved against baseline
- where the causal effect showed up across the network
- whether the result looks like real semantic leverage or shallow disruption

If that works, the artifact layer becomes the bridge from theory to architecture. You stop arguing from vibes and start arguing from replayable evidence.

## Constraints
- Reuse the existing research spine in [`consciousness_loop/artifacts.py`](/Users/peterwei/wokspace/abstractor/consciousness_loop/artifacts.py), [`consciousness_loop/self_kv_bias_tournament.py`](/Users/peterwei/wokspace/abstractor/consciousness_loop/self_kv_bias_tournament.py), and [`consciousness_loop/self_kv_bias_signal.py`](/Users/peterwei/wokspace/abstractor/consciousness_loop/self_kv_bias_signal.py).
- Keep the diff minimal. Extend the current artifact and history model instead of introducing a parallel analytics subsystem.
- Preserve offline-friendly local execution. Stronger-model judging can remain optional, but artifact generation and local summaries should work without network services beyond the chosen model runtime.
- Treat current JSONL history files as useful caches and indexes, not as the canonical scientific record.
- Avoid overfitting the mechanistic story to a single metric such as `late_to_early_ratio`.

## Premises
1. The first product is a falsifiable measurement system for the self-vector hypothesis, not the hypothesis itself.
2. Existing tournament, signal, and artifact paths already solve most of the plumbing and should be extended, not replaced.
3. A good mechanistic score must capture causal layer profiles across early, middle, and late bands, because semantically useful influence may not live only in the last layers.
4. A checkpoint is only scientifically useful if someone can inspect why it won without rerunning the model.

## Approaches Considered
### Approach A: History-First Patch
Summary: Keep JSONL history files as the main record and add more fields for scoring, judge metadata, and layer summaries.

Effort: S
Risk: Medium

Pros:
- Minimal change to the current code paths
- Fastest route to more metrics in the CLI
- Reuses existing append-only history semantics

Cons:
- Provenance stays fragmented across history rows, artifacts, and cached rounds
- Replay and checkpoint comparison remain awkward
- Easy to accumulate duplicated logic between tournament history and signal history

Reuses:
- `TournamentHistoryStore`
- `append_signal_history()`
- Existing round cache keys

### Approach B: Artifact-First Research Spine
Summary: Introduce a versioned `research_run` artifact as the canonical record for a candidate evaluation, then make JSONL history files lightweight indexes and cache logs that point back to artifact paths.

Effort: M
Risk: Low

Pros:
- Gives each checkpoint a single replayable source of truth
- Makes benchmark claims inspectable without rerunning the model
- Fits the repo's existing versioned artifact pattern
- Makes future replay, stronger judging, and comparison UIs much easier

Cons:
- Requires careful schema design up front
- Forces some refactoring in tournament and signal reporting paths
- Needs migration logic for older artifacts and legacy history rows

Reuses:
- `build_versioned_artifact_payload()` and `write_json_artifact()` in [`consciousness_loop/artifacts.py`](/Users/peterwei/wokspace/abstractor/consciousness_loop/artifacts.py)
- Promotion artifact payloads in [`consciousness_loop/self_kv_bias_tournament.py`](/Users/peterwei/wokspace/abstractor/consciousness_loop/self_kv_bias_tournament.py)
- Existing signal benchmark calculations in [`consciousness_loop/self_kv_bias_signal.py`](/Users/peterwei/wokspace/abstractor/consciousness_loop/self_kv_bias_signal.py)

### Approach C: Benchmark Card Overlay
Summary: Leave the generation pipeline mostly intact and add a summarizer that reads current history and artifact outputs into a single comparison report.

Effort: M
Risk: Medium

Pros:
- Safer if the current output formats are still in flux
- Good for presentation and quick iteration
- Can be built with little disruption to live experiment code

Cons:
- Does not fix missing provenance at the source
- Keeps the hard logic spread across multiple files and payload types
- Risks becoming a reporting veneer over weak measurements

Reuses:
- Existing history rows
- Promotion artifacts
- Existing CLI output summaries

## Recommended Approach
Choose Approach B, artifact-first research spine.

This is the smallest change that makes the results scientifically legible. It matches the repo's current direction toward versioned artifacts, keeps the architecture boring, and turns history rows into indexes instead of pretending they are the whole experiment. The artifact becomes the thing you compare, replay, judge, and promote.

## Proposed Artifact Shape
Add a new artifact kind, `research_run`, with a schema version independent from fork-run and coevolution cohort artifacts.

The artifact should contain:
- top-level metadata: `artifact_kind`, `schema_version`, `artifact_path`, `created_at`
- repo/runtime config: model, judge model, backend, self dimension, prompt-bank fingerprint, prompt count, seed, projection scale, rollout caps, selection weights
- candidate identity: self-vector label, self-vector signature, source artifact reference, genome parameters, branch/slot when loaded from another artifact
- baseline identity: frozen baseline genome and relevant comparison settings
- mechanistic summary:
  - per-prompt early/middle/late band summaries
  - late, middle, and early KL aggregates
  - rollout KL aggregates
  - decoded divergence and top-k overlap aggregates
  - warnings when effect concentrates in clearly shallow or unstable regions
- behavioral summary:
  - judge win rate
  - selected win rate
  - bootstrap confidence interval
  - judge score delta
  - selection score delta
  - Elo deltas
- verdict summary:
  - behavioral pass
  - mechanistic pass
  - strong signal pass
  - explanatory notes listing which thresholds were met or failed
- compact evidence:
  - one row per prompt with prompt fingerprint, winner, judge scores, selection scores, and causal band metrics
  - references to heavier cached round artifacts if needed later

Do not store full raw logits or giant layer tensors in the `research_run` artifact. That turns each run into a landfill. Store compact summaries plus stable references to cache entries when deep forensics are needed.

## Mechanistic Scoring Rules
The scoring model should move away from a single heroic metric.

Use a composite mechanistic signal:
- require non-trivial causal effect above a floor
- require acceptable rollout stability under a cap
- compare early, middle, and late band behavior explicitly
- compute whether the strongest useful effect lands in middle or late bands rather than rewarding only "late beats early"

Recommended interpretation:
- `bad`: effect is mostly early-band disruption, rollout instability is high, or decoded divergence grows without behavioral gains
- `interesting`: middle or late bands move with moderate stability, but behavioral gains are not yet reliable
- `strong`: behavioral gains clear threshold and confidence interval, while middle/late causal profiles stay above floor and rollout instability stays below cap

This keeps the metric attached to the architectural claim instead of letting one ratio become a superstition.

## Open Questions
- Should `research_run` artifacts include a slim transcript excerpt for the most decisive prompts, or stay purely numeric and refer back to cached rounds for text?
- Should the mechanistic verdict use fixed thresholds from day one, or support model-specific thresholds keyed by model family?
- How much backward compatibility is worth carrying for older promotion artifacts that do not include the new summary fields?

## Success Criteria
- A single saved artifact is enough to explain why a candidate passed or failed without rerunning the benchmark.
- Running the signal benchmark writes one canonical `research_run` artifact plus a lightweight history row pointing to it.
- Two checkpoints can be compared offline using only saved artifacts and prompt-bank fingerprints.
- Mechanistic summaries report early, middle, and late band behavior, not just a single `late_to_early_ratio`.
- The artifact format is small enough to keep many runs, stable enough to replay later, and explicit enough to support future replay or judge extensions.

## Distribution Plan
This feature ships as part of the existing Python package and CLI entry points. No new external artifact type or publish pipeline is needed. The current `pyproject.toml` and editable-install workflow remain the distribution channel.

For release discipline:
- keep artifact generation behind the existing CLI commands in `consciousness_loop`
- document the new artifact kind and output path in [`README.md`](/Users/peterwei/wokspace/abstractor/README.md)
- rely on the existing test suite plus CI already used for the repo; no extra deployment layer is required
