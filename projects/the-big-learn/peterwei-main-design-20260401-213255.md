## Problem Statement

The Big Learn should not become a generic AI Chinese tutor.

The core product is a host-native guided-reading ritual for Chinese classics inside Codex and Claude Code. The user should experience The Big Learn from within those confines first, as a thread-native skill or command, not as a separate application they conceptually enter after setup. The ritual should be serious enough for advanced or highly motivated readers who want to understand classical texts with AI support, while still being survivable for a beginner who does not know where to start.

The product protocol is the center of gravity:

1. read through the line
2. write questions or comments as they arise without breaking the reading pass, letting the AI work on them in the background while you keep moving through the text
3. do a personal translation
4. discuss the translation
5. write a response
6. discuss the response
7. move to the next line

The design problem is how to make that ritual feel coherent, useful, and alive for different learner types inside the constraints of a coding-assistant thread without collapsing back into mushy chatbot tutoring.

## What Makes This Cool

- The source material has gravity. Reading `Da Xue`, `Lunyu`, or `Daodejing` feels different from grinding phrase drills.
- The same ritual works as both beginner on-ramp and advanced practice. That is much cleaner than maintaining two products.
- The assistant behaves less like a quiz app and more like a seminar partner that keeps continuity intact.
- The char-by-char breakdown and character exploder create an "I can actually get unstuck here" moment for beginners.
- Stored learner artifacts can make the thread feel like a private tutor with memory instead of a one-shot chat.

## Constraints

- Stay host-native. Do not design a standalone app as the primary surface.
- Imagine the product from inside Claude Code or Codex. Menus, artifacts, prompts, persistence cues, and progress should all make sense within a thread-first assistant UX.
- Keep the reading protocol intact. Helpful features cannot shatter continuity.
- Preserve classics-first positioning. The text is not interchangeable content.
- Reuse the existing repository split: host adapters in `hosts/`, workflow logic in `workflows/`, support runtime in `the_big_learn/`, state in progress/flashcard artifacts.
- Support both local six-layer annotation and raw-source reading modes honestly.
- Treat the current curriculum as bundled and immediately readable from local package data. Do not design first-run curriculum access around live discovery.
- Keep all user-facing guidance in English while staying grounded in the Chinese source text.
- The current `git clone` plus setup flow is supporting infrastructure for builders. It is not the product starting point in the design.

## Premises

1. The product is advanced-core. It is fundamentally a deep-reading system for classics, not a broad beginner-first language app.
2. Guided reading itself is the beginner on-ramp because the line scaffold, deferred-question protocol, and character exploder make entry possible without creating a separate "beginner mode."
3. The primary UX is one host-native ritual inside Claude Code or Codex, not a mobile app and not a generic chatbot.
4. Classical texts are part of the hook, not just the curriculum.
5. Character explosion should be prominent and proactive, but governed by continuity-preserving trigger rules rather than firing indiscriminately.
6. The system should persist artifacts that describe the learner's style and friction points so scaffolding can adapt over time.
7. The requested product change belongs in the workflow and host-prompt layer first, with runtime/state support underneath it.

## Approaches Considered

### Approach A: One Ritual, Proactive Scaffolding

Keep a single guided-reading ritual for everyone. Provide heavy line scaffolding, char-by-char breakdowns, and proactive character-exploder suggestions whenever the model detects likely confusion.

Pros:
- Cleanest product story
- Fastest path to a believable experience
- Keeps beginner and advanced readers in one system

Cons:
- Easy to over-fire the exploder and destroy flow
- Can feel too chatty for advanced readers
- Requires careful prompt discipline even without persistent user modeling

### Approach B: One Ritual, Adaptive Scaffolding

Keep the same guided-reading ritual for everyone, but adapt the level and timing of support using stored learner artifacts. Beginners or stuck readers get more exploder behavior and explicit scaffolding. Readers who want less unsolicited help should be able to choose a quieter, more text-led posture without leaving the core ritual.

Pros:
- Best match for the product truth that emerged in the session
- Lets one ritual serve both the beginner on-ramp and serious readers
- Makes memory and progress artifacts materially useful

Cons:
- More state and behavior tuning
- Harder to explain than a fixed behavior profile
- Requires explicit policy for when adaptation is allowed to change the flow

### Approach C: Trojan Horse Distribution Layer

Treat the real product as a delightful reading companion that happens to run inside Claude Code or Codex, and invest early in packaging, install, and first-run copy that hides the power-user substrate.

Pros:
- Most surprising long-term wedge
- Sharp answer to why this belongs in agent tools
- Strong distribution story if it works

Cons:
- Risks spending too much time on packaging before the ritual itself is proven
- Not well matched to the current builder install path
- Adds complexity outside the core learning loop

## Recommended Approach

Choose Approach B: One Ritual, Adaptive Scaffolding.

This preserves the cleanest insight from the session: guided reading is both the product and the beginner on-ramp. There is no second onboarding product. The real system is the same seminar-like flow for everyone, but the intensity and timing of help changes as the assistant learns how this reader gets stuck.

It also respects the product boundary you clarified late in the session: the app should be imagined from inside Claude Code or Codex. That means the design should optimize for thread behavior, compact menus, visible saved artifacts, and clean transitions between reading, discussion, and recall without depending on separate app chrome.

Recommended behavior shape:

- Default to one shared protocol: line, optional deferred question or comment, personal translation, discussion, personal response, discussion, next line.
- Open as a host-native guided-reading command or skill, with compact book and chapter menus that feel natural in a thread.
- Assume that every book in the current curriculum can open from bundled local data without a web prerequisite.
- Keep the char-by-char table visible on supported lines.
- Make the character exploder prominent and easy to invoke.
- Keep saved artifacts visible in host-native ways: short status tags, recap blocks, and references to prior translations or responses rather than dashboard-style UI assumptions.
- Allow proactive exploder suggestions only when the line contains a likely blocking character, semantically dense moral/political vocabulary, or a pattern the learner has repeatedly struggled with.
- Cap proactive exploder suggestions to one or two characters at a time so the product does not collapse into constant interruption.
- Persist learner-style artifacts such as "asks conceptual questions before grammar," "benefits from decomposition on virtue terms," or "prefers deferred answers" and use them to tune the thread later.
- Expose one persistent reading-style switch with two names: `Proactive` and `On-Demand`.
- `Proactive` increases prompt explicitness, volunteered help, exploder readiness, and feedback visibility for readers who want the assistant to initiate support more often.
- `On-Demand` keeps the assistant more text-led and reactive, waiting for the learner to ask before surfacing most extra help, but it does not remove any capability the learner can still request.
- The reading-style switch tunes proactivity, prompt explicitness, discussion length, and exploder behavior.
- The reading-style switch does not change the ritual order, persistence behavior, or the seriousness of textual engagement.
- Learner-style artifacts may continue adapting behavior inside the selected style, but they should not silently override the learner's named choice.

## Thread Information Architecture

The host thread should have a clear top-to-bottom hierarchy so returning readers can resume immediately without losing the curriculum context and first-time readers can still see the full shape of the product.

Opening-turn hierarchy:

1. `Resume where you left off` block, shown only when saved work exists.
2. `Recommended next step` block, usually `Start with Da Xue` for first-time readers or `Continue Chapter N` for returning readers.
3. Full curriculum menu.
4. One-sentence instruction for how to choose, by number or title.

The resume block should be short and operational, not sentimental. It should show:

- current book and chapter
- whether `Personal Translation` is saved
- whether `Personal Response` is saved
- one clear action: continue, reopen chapter menu, or switch books

The opening turn should never auto-resume silently. Resume is the first thing visible for returning readers, but the full curriculum menu still appears below it so the learner retains orientation and agency.

The emotional tone of the opening turn should feel like a seminar invitation, not like onboarding copy and not like a terminal command reference. The product should assume the learner is serious enough to be here, while still helping them feel that the text is enterable.

Opening-turn voice rules:

- one calm sentence on what guided reading is
- one practical sentence on how to choose a book or continue
- no motivational fluff
- no apology for difficulty
- no beginner-labeling language such as `don't worry if you're new`
- no wall of instructions before the learner sees the menu

The curriculum menu should remain a single flat numbered list rather than grouped lanes. The recommendation and resume blocks do the guidance work above the list. Inside the list itself, each book should get one short English framing line so the learner can scan the difference between titles without needing prior literary context.

Book-menu rules:

- keep all supported books visible in the first turn
- keep numeric ordering stable across hosts
- place the Chinese title and English transliteration on the main line
- place one short English framing line directly underneath, explaining what kind of text it is or where it sits in the curriculum
- include a compact support-level label in the framing line or directly after it
- keep saved-work tags compact and secondary, never louder than the title
- do not add paragraph-length summaries inside the menu

First-turn layout sketch:

```text
[Resume where you left off]
Da Xue, Chapter 3
[translation saved] [response saved]
Continue reading | Reopen chapter menu | Switch books

[Recommended next step]
Start with Da Xue if you want the designed on-ramp into the classics.

[Curriculum]
1. Da Xue
   The designed starting point, short, dense, and heavily supported.
2. Zhong Yong
   Continues the Four Books path with denser moral and metaphysical language.
3. Lunyu
   Aphoristic sayings and conversations, broader but less linear.
...

Reply with a number or a book name.
```

Suggested first-turn tone:

```text
Guided reading keeps the text, your questions, and your own translation in one continuous thread.
Choose a book by number or title, or continue where you left off.
```

Chapter-selection hierarchy:

1. Chapter title and short English framing.
2. Support-level label for that chapter.
3. Saved-work tags for that chapter.
4. Length signal, either line count or approximate character count when available.
5. Chapter menu choices.

The chapter menu should stay chapter-shaped, not excerpt-shaped. Local annotated spans such as `Opening outline` may appear as progress metadata or coverage notes, but they must not replace the full book chapter list.

Bundled-versus-live state rules:

- For the current curriculum, the opening book menu and chapter menu should assume local availability and should not warn about web requirements.
- The normal learner path should never hit a source-discovery blocker while opening a bundled curriculum book.
- Live-source selection is a secondary path only for alternate editions, explicit source changes, or texts outside the bundled curriculum.
- When live-source mode is used, the thread should say so plainly so the learner understands why the reading surface may have generated glosses instead of stored six-layer annotations.

Support-level visibility rules:

- Every book and chapter entry should communicate one of three states in compact English: `stored annotation`, `bundled text + generated help`, or `alternate source`.
- `stored annotation` means the learner can expect the full six-layer reading surface on supported lines.
- `bundled text + generated help` means the text is local and immediately readable, but some support is generated during the session instead of fully pre-encoded.
- `alternate source` should appear only when the learner explicitly chooses a different edition or a text outside the bundled curriculum.
- When a chapter mixes stored annotation coverage and bundled raw-source coverage, the chapter framing should say so before line rendering begins.
- If the reading mode changes inside a chapter, for example when the annotated span ends and bundled raw-source support takes over, the thread should insert one short mode-change note before the next line rather than silently changing the surface.
- These labels should stay compact and factual. They are trust markers, not warnings.

Reading-turn hierarchy:

1. Current line in Chinese.
2. Current line in English.
3. Char-by-char table when supported.
4. Optional exploder cue only when the line is likely to block progress.
5. Learner action prompt: write a question or comment if needed, then keep reading or attempt translation at the appropriate moment.
6. Discussion turn.
7. Personal response prompt.
8. Response discussion, then move on.

Reading-shell rules:

- Each line should reuse one compact recurring shell so the learner quickly recognizes the rhythm without feeling trapped inside a worksheet.
- The Chinese line is the visual anchor and should remain the loudest element in the turn.
- The full-line English gloss and char-by-char table do most of the structural work. Additional headings should be used sparingly.
- Avoid repeating large section labels such as `Chinese`, `English`, `Breakdown`, `Your Translation`, and `Feedback` on every line unless the host genuinely needs them for clarity.
- Prefer spacing, ordering, and one or two stable cue phrases over a stack of bold labels.
- The assistant's own prose should stay shorter than the learner-facing text block whenever possible.
- Translation feedback and response discussion should read like a seminar exchange, not rubric grading.
- Proactive exploder behavior must never block the next reading prompt.

Compact line-shell sketch:

```text
大学之道，在明明德，在亲民，在止于至善。
The way of great learning lies in making bright virtue shine...

[char-by-char table]

If a question comes up, write it down and keep reading.
Your translation?
```

## AI Slop Risk

The main slop risk in this product is not visual chrome. It is host-thread bureaucracy: too many repeated headings, too many status blocks, too much assistant copy, and a reading experience that starts to feel like a generated worksheet instead of a live encounter with a text.

Thread-specific anti-slop rules:

- Do not turn every line into a seven-part template with loud labels.
- Do not let assistant instructions become longer than the source text being studied.
- Do not restate the same ritual directions on every turn once the learner is already in the flow.
- Do not bury the Chinese line under status tags, warnings, or helper prose.
- Do not make every support action a named feature block.
- Do not present translation feedback like a classroom rubric with numbered deductions unless the learner explicitly asks for that mode.
- Do not inflate chapter-end recaps into essay-length summaries before the learner has given their own response.
- Do not let a proactive exploder stall the line flow while heavy character-analysis content is being prepared.

The design target is a compact reading shell with a few meaningful repeated forms:

- opening menu
- chapter menu
- line shell
- chapter-end recap
- save or retry state when needed

Everything else should feel like conversation in service of those forms, not like additional UI furniture.

Non-blocking exploder rule:

- When the system detects a likely blocking character or term, it may start preparing exploder content in the background during the reading pass.
- During that pass, show at most one compact cue that extra character help is being prepared or is available.
- Do not hold the next learner prompt, translation step, or next line on exploder latency.
- Reveal proactive exploder output only when it is ready and only in a compact form that sits underneath the current line shell.
- Limit proactive exploder output to one or two characters at a time.
- If exploder content is still not ready by the time the learner wants to move on, keep reading moving and surface the prepared result later at the learner's stopping point or when they explicitly ask for it.

## Design System Alignment

This review now creates a repository-wide [`DESIGN.md`](/Users/peterwei/wokspace/the-big-learn/DESIGN.md) for thread-native surfaces. The plan-level decisions below should align to that file so Codex and Claude do not drift in vocabulary or rhythm.

Thread-surface vocabulary:

- Saved-work tags use compact bracketed labels such as `[translation saved]`, `[response saved]`, and `[unsaved]`.
- Support-level labels stay in plain English: `stored annotation`, `bundled text + generated help`, `alternate source`.
- Primary action lines stay short and imperative: `Continue reading`, `Retry save`, `Choose a chapter`, `Your translation?`, `Your response?`
- Secondary actions should be visibly lower-emphasis and never outnumber primary actions.
- Tone stays English-first for guidance and operations, with Chinese shown for source text, line content, and character-level support.
- Avoid emojis, decorative glyphs, and marketing-style headings in the reading flow.
- Use consistent cue phrases across hosts so the ritual feels like the same product even when installed into different assistants.

Host-alignment rules:

- Codex and Claude should share the same saved-tag vocabulary, support-level labels, and chapter-end cue phrases.
- Differences between hosts should come from host mechanics, not from different product language.
- Any new recurring thread token introduced during implementation should be added back into `DESIGN.md` rather than invented ad hoc in one host only.

## Responsive and Accessibility

This is a thread-native product, so responsive behavior means adapting to narrow host panes, large text settings, and uneven markdown rendering. Accessibility means the learner can still follow the reading structure when visual table formatting degrades.

Responsive rules:

- Default to the five-column char-by-char table when the host surface renders it clearly.
- If the host pane is narrow or table readability becomes poor, switch to a stacked per-character list instead of forcing a cramped table.
- The stacked fallback should preserve the same information order: character, reading, character translation, phrase, phrase translation.
- The opening menu, chapter menu, and chapter-end recap should avoid multi-column assumptions entirely and remain readable as plain vertical blocks.
- Progress tags and support-level labels should stay short enough to wrap cleanly without dominating the line.

Stacked fallback sketch:

```text
德
Reading: de2
Char translation: virtue
Phrase: 明德
Phrase translation: bright virtue
```

Accessibility rules:

- Preserve a stable top-to-bottom reading order so the thread still makes sense without visual scanning.
- Keep recurring cue phrases consistent so a screen-reader or text-only learner can build a reliable mental model of the flow.
- Do not rely on spacing alone to distinguish source text, helper content, and action prompts.
- Keep action prompts short and explicit, for example `Your translation?` or `Retry save`.
- Avoid decorative Unicode or punctuation-heavy separators that make assistive reading noisy.
- When switching from table to stacked mode, do so completely for that rendered line rather than mixing both formats in one confusing turn.
- Default to compact prompts in the normal reading flow, but escalate to more explicit repeated cues when the host surface is visually rough, the learner prefers stronger structure, or accessibility needs make that helpful.
- When explicit cue mode is used, keep the repeated prompts stable across turns so the added redundancy improves orientation instead of adding noise.

## Interaction States

The plan must specify what the learner sees in normal, empty, partial, and failure states. In a host-thread product, these states are the interface. If they are underspecified, the implementation will feel random and untrustworthy.

| Feature | Loading | Empty | Error | Success | Partial |
|---------|---------|-------|-------|---------|---------|
| Opening turn | Brief orientation line while progress is read | No resume block, show `Recommended next step` plus full curriculum menu | Explain that saved progress could not be read, continue with curriculum menu, and avoid pretending prior work is loaded | Resume block appears when saved work exists | Resume block appears but omits unavailable metadata rather than inventing it |
| Book menu | Not applicable beyond brief thread transition copy | All current curriculum books still visible even when no saved work exists | Only use an error state for explicit alternate-source requests outside bundled curriculum | Chosen book resolves immediately from bundled local data, with support-level labels visible in the list | If a book has bundled raw source but not local six-layer annotation, say so plainly in its framing line |
| Chapter menu | Brief transition copy while chapter metadata loads | Show chapter list with no saved tags when nothing has been saved yet | Explain if chapter metadata cannot be read and offer retry plus switch-book action | Chapter list with support-level label, saved-status tags, and length signal | Show coverage notes when a chapter mixes stored annotations and raw-source support |
| Reading turn | Short line-level transition only when rendering a new unit | Not applicable once a chapter is opened | If a line cannot render, stop and explain which line failed rather than skipping silently | Full line, char-by-char support when available, learner prompt, and next-step discussion | Raw-source lines may show generated gloss support instead of stored six-layer support, but the distinction must stay visible |
| Chapter-end save | Brief `Saving your work...` line after chapter-end response is ready | Not applicable | Keep all unsaved learner work visible, mark the chapter as `unsaved`, offer immediate retry, and do not claim the recap artifacts are stored yet | Explicit save confirmation after persistence succeeds | If some auxiliary artifacts fail but core chapter artifacts save, state exactly what saved and what still needs retry |
| Deferred question review | Brief transition line before review starts | Say `No deferred questions or comments from this pass` and offer next chapter or stop | If deferred notes cannot load, say so plainly and keep chapter-end artifacts visible | Return to each deferred question or comment with its linked line context | If only some deferred notes can be revisited, answer what is available and mark the remainder as still pending |

Chapter-end save failure behavior:

- Keep the full `Personal Translation` and `Personal Response` visible in-thread after any failed save attempt.
- Show an explicit `unsaved` state block directly under the recap rather than burying the failure inside prose.
- Offer one primary action, `Retry save`, and one secondary action, `Stop here and keep this thread open`.
- Do not continue into the next chapter automatically while the current chapter remains unsaved.
- Do not announce saved learner-style artifacts, generated annotations, or deferred-question state unless the persistence command actually succeeded.
- Once a retry succeeds, replace the `unsaved` state block with a short save confirmation and then continue to deferred-question review.

Example save-failure block:

```text
[Save did not complete]
Your chapter translation and response are still visible in this thread, but they are not stored yet.
Primary action: Retry save
Secondary action: Stop here and keep this chapter open
```

## User Journey and Emotional Arc

The guided-reading thread should feel different across three moments:

1. First 10 seconds: the learner feels oriented, not overwhelmed.
2. First 5 minutes: the learner feels they are genuinely reading the text, not using a toy tutor.
3. After several sessions: the learner feels the thread remembers how they learn and respects their pace.

Storyboard:

| Step | User does | User feels | Plan support |
|------|-----------|------------|--------------|
| 1 | Opens guided reading for the first time | Curious but slightly intimidated | Seminar-style opening tone plus a visible recommended start |
| 2 | Sees the curriculum menu | Oriented, aware there is a path | Flat numbered menu with one-line English framing under each title |
| 3 | Opens a chapter | Reassured that this is readable in small pieces | Chapter framing, support-level label, saved-status tags, and length signal |
| 4 | Sees the first line and char-by-char support | Relief, `I can actually get into this` | Fixed line-first hierarchy and visible decomposition support |
| 5 | Has a question or comment without wanting to lose momentum | Safe to keep moving | Deferred-question discipline keeps reading continuous and postpones consumption of answers |
| 6 | Attempts a translation | Exposed but engaged | Discussion tone should critique the reading without shaming the learner |
| 7 | Finishes a chapter and writes a response | Ownership, seriousness, and closure | Chapter recap, chapter-end response, dual-lens evaluation, and explicit save state |
| 8 | Returns later | Continuity and respect | Resume block, saved artifacts, and learner-style adaptation without fake familiarity |

Tone by phase:

- Opening turn: invitational, calm, serious.
- Line work: attentive and concrete, with the text louder than the assistant.
- Translation feedback: candid but not punishing.
- Chapter-end discussion: slightly more reflective, because the learner has now earned a broader conversation.
- Save confirmation and resume cues: terse and operational.

The first successful moment should happen early. By the first or second rendered line, the learner should feel that difficult classical Chinese can still be entered through the combined effect of the full-line gloss, the char-by-char table, and the permission to keep reading before answering every question.

Deferred-question design rule:

- Do not design a visible queue widget, queue counter, or sidebar-like queue section.
- Treat deferred questions and comments as part of the participant's reading discipline, not as a separate product feature.
- The assistant may acknowledge that it will come back to a question, but the acknowledgment should be compact and should not compete with the text.
- The important design promise is continuity: the learner can write a question when it arises, keep reading in one pass, and return later without losing context.
- During the reading pass, the system may prepare answers silently in the background, but it should not reveal those answers inline before the learner reaches the chosen stopping point.
- Review later should feel fast because the work has already been prepared, not because the learner was shown spoilers during the reading pass.

ASCII flow:

```text
Open guided reading
  -> Resume block (if saved work exists)
  -> Recommended next step
  -> Full book menu
  -> Full chapter menu
  -> Line-by-line reading loop
  -> Chapter recap and saved artifacts
  -> Deferred-question review
  -> Next chapter or stop
```

## Success Criteria

- A learner already inside Claude Code or Codex can understand how to start a guided-reading session immediately from the host-native prompt flow.
- A first-time motivated builder can install the repo and enable that guided-reading session in under 10 minutes using the current builder path.
- The line protocol remains consistent: deferred-question opportunity first, personal translation next, discussion, personal response, discussion, then continue.
- Questions and comments written during the reading pass are actually deferred by default and then revisited later without losing line ids or context.
- Deferred answers can be prepared silently during the reading pass, but they are only revealed when the learner reaches the chosen stopping point or explicitly asks to review them.
- The system persists `Personal Translation`, `Personal Response`, deferred question context when stored, generated annotations, and learner-style artifacts across sessions.
- On re-read, the assistant can use saved artifacts to adapt support without hallucinating stored structure that does not exist.
- Character explosion becomes more helpful over time, not noisier. A good default is no more than one or two proactive suggestions per line.
- The learner can explicitly choose between `Proactive` and `On-Demand`, and that choice persists across sessions.
- `Proactive` and `On-Demand` change support intensity and prompt style without changing the underlying reading ritual.
- The chapter-end recap includes full English translation, full Chinese text, the learner's stitched personal translation, and deferred-question review.
- The experience does not assume a separate web app, dashboard, or side panel in order to feel complete.

- Keep distribution automation tied to host asset integrity so a release cannot ship with broken prompts or missing bundled source data.
