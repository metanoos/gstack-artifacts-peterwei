# Test Plan
Generated by /plan-eng-review on 2026-03-27
Branch: main
Repo: geometer-jones/synchrono-city

## Affected Pages/Routes
- `/app` — verify startup loads relay identity, fetches nearby beacons from relay NIP-29 metadata, and renders markers from beacon records instead of Concierge social bootstrap data.
- `/app?beacon=<id>` — verify opening a beacon loads beacon-scoped notes by `h` tag, merges Concierge study-circle overlay fields, and renders the correct card/thread state.
- `/app/settings` — verify room-policy/admin forms now use `beacon:<id>` room ids instead of legacy `geo:<operator>:<geohash>` ids.

## Key Interactions to Verify
- World startup performs NIP-11 fetch, NIP-29 beacon discovery by `g` prefix, and tiny Concierge control-plane bootstrap without split-brain data.
- Clicking a beacon marker opens that beacon without auto-joining media.
- Clicking `Join call` requests a LiveKit token for `beacon:<id>` and the global overlay connects correctly.
- Posting a beacon note publishes a kind `1` event scoped by `h=<beacon-id>`.
- Relay-unavailable startup shows an honest retryable empty state rather than stale or fake beacon data.

## Edge Cases
- Relay returns malformed `39000` metadata missing `d` or required display tags.
- Concierge overlay data is missing for a valid relay beacon.
- Relay startup requests fail while Concierge control-plane request succeeds.
- Room permission/admin UI still contains legacy `geo:` ids anywhere in the flow.
- Beacon note query returns events without the expected `h` tag.

## Critical Paths
- App load -> relay capabilities load -> nearby beacons render on the map.
- User opens beacon -> thread loads -> study-circle overlay fields render.
- User joins call from beacon -> token request uses `beacon:<id>` -> call overlay becomes active.
- Admin updates beacon room policy -> stored room id matches `beacon:<id>`.
