## Key Interactions to Verify
1. Empty-map click opens `Light Beacon`, then the creation form, and create-or-return-existing resolves to one beacon.
2. Marker selection remains safe to inspect and never auto-joins live media.
3. Joining from beacon detail requests the correct room namespace and keeps the detail state intact on failure.
4. Publishing from World writes beacon-scoped public conversation while keeping Pulse reserved for profiles and non-beacon public context.
5. Pulse profile navigation from beacon UI still works after beacon-thread exclusion lands.
6. Rollout compatibility allows old readers and new writers to coexist during cutover.

## Edge Cases
- Backend still returns legacy `geo:` room id during migration.
- Relay queries still return legacy `g`-scoped notes while new writes emit `h` scope.
- Duplicate beacon creation race between two clients.
- Token vending returns 401/403 for the new room namespace.
- Nostr signer missing or relay rejects beacon-scoped publish.
- Refresh fails after cached World state is already visible.
- User is already in one call and inspects another beacon.

## Critical Paths
- Compatibility read path: read `h` + legacy `g` during rollout, then remove legacy support.
- Compatibility room path: accept or translate `beacon:<geohash8>` end-to-end before deleting `geo:` fallback.
- Route boundary: Pulse must not surface beacon-thread content once beacon scope is canonical.
