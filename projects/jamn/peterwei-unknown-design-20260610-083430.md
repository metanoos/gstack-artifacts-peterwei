## Problem Statement

Every editor throws the writing process away. The finished document survives; the three deleted openings, the paragraph that sat half-done over a weekend, the sentence retyped eleven times — gone. The builder wants a text editor "like Google Docs but with replay": every input recorded with its timing, so any document can be played back. Large gaps between sessions are indicated honestly in the replay (labeled with their true length) but compressed in playback so the replay stays watchable. Replay time is narrative time, not wall-clock time.

Three audiences for a replay, all chosen: (1) the writer — a private mirror of their own process; (2) readers — publish a piece together with its replay, a new media format; (3) skeptics — the replay as soft evidence a human wrote it. Collaboration was explicitly deferred to v2.

## What Makes This Cool

- **The gap-compression insight (the builder's own, stated in the first sentence):** a three-day pause becomes a visible "3 days later" beat. That single move turns a log viewer into a film of the document being written.
- **The inversion of an entire category.** The existing "writing replay" tools (Draftback, Revision History, GPTZero replay, Passed.ai, WriteTrace, Process Feedback) are all teacher-side anti-AI forensics bolted onto Google Docs' coarse revision history. This is the writer-owned inverse: native keystroke fidelity, recorded for yourself, published proudly. Nobody prominent builds this side.
- **The finished essay becomes the last frame, not the product** (Codex's framing). Deleted passages, struggle heatmaps, and "watch this paragraph get born" clips are all latent in the same event log.
- **One data model, four products:** current text, version history, replay, and provenance are all projections of one append-only event log.

## Constraints

- **Daily driver, not a demo.** The builder twice overrode "ship the minimal version" recommendations (D4, D8). This is the tool they intend to write everything in. Trust is the bar: autosave that never loses a keystroke, multi-device, durable backup.
- **Plain text / markdown only in v1.** No rich text, no comments, no embeds. The editor earns trust before features.
- **Replay must be watchable.** Session gaps compressed with honest labels; in-session pauses capped (~2s at 1x); a "skim" speed so a long doc replays in about a minute.
- **Collab deferred to v2**, but every event carries an author id from day one so the data model never blocks it.
- **Recording is always on** — that is the point of daily-driver scope — but the log is private by default; publishing a replay is an explicit per-document act.
- Solo builder working with Claude Code; effort estimates assume that leverage.

## Premises

1. **Replay is a writer-owned medium, not forensics.** The writer records for themselves and chooses to publish; nobody demands the log. (Inverts the existing category.)
2. **The event log IS the document.** The visible doc is a projection of an append-only, timestamped edit log. One source of truth; text, history, replay, and provenance all derive from it. (The one one-way door in this design.)
3. **Native keystroke fidelity is the moat.** Recording in the editor at true timing beats every Google Docs bolt-on, which reconstructs from coarse snapshots after the fact.
4. **Replay time is narrative time.** Gaps between sessions are shown honestly (true length labeled) but compressed in playback; pauses within a session play near-true, capped.
5. **Daily driver means trust-first, features-second.** v1 = plain text/markdown, autosave that never loses a keystroke, a doc list. The bar is "I trust it," not "it matches Docs."
6. **Collab is v2 — but never precluded.** Events carry an author id from day one, even while there is only one author.
7. **Provenance is soft.** The replay is persuasive evidence a human wrote it, not cryptographic proof. (Supported by the 2026 arXiv result "On the Insecurity of Keystroke-Based AI Authorship Detection": keystroke logs show a human typed, not that the human originated the text.)

All seven premises confirmed by the builder; Codex's independent read challenged none of them.

## Approaches Considered

### Approach A: The Kernel — local-first web app (minimal viable)
CodeMirror 6 editor, append-only event log in IndexedDB, doc list, replay viewer with gap title-cards, publish = export a static JSON replay bundle.
- **Effort:** S (human: ~2 weeks / CC: ~2–3 days) | **Risk:** Low
- **Pros:** First "whoa" within days; zero backend; the event schema gets battle-tested cheaply.
- **Cons:** Single browser profile — storage eviction can destroy a daily driver's only copy; publishing is manual file hosting.
- **Reuses:** CodeMirror 6, Dexie/IndexedDB.

### Approach B: The Daily Driver — kernel + thin sync-and-publish server (CHOSEN)
Same client as A, plus a small server (Bun/Node + SQLite or Postgres) doing exactly two things: append-only event sync (multi-device, durable backup) and hosting published replays at clean URLs. Magic-link auth. Publish = flip a flag.
- **Effort:** M (human: ~1–2 months / CC: ~1–2 weeks) | **Risk:** Med (one real CS problem — multi-device merge, see the merge policy in Recommended Approach — plus ops)
- **Pros:** Delivers the actual D4 choice — write everywhere, never lose a keystroke; replay links are real URLs from v1.
- **Cons:** A server to run and pay for; auth/accounts are boring work before new fun.
- **Reuses:** Everything from A unchanged — the server only ships events.

### Approach C: Nostr-Native — replays as signed events (creative/lateral)
Writing sessions chunked into signed Nostr events on relays; a doc is an addressable event stream under the builder's npub. Sync, backup, publishing, identity, and provenance signatures come from the protocol. Would share infra/identity with the prior Jamn chat design and use the NDK checkout already in the workspace.
- **Effort:** L (human: ~2–3 months / CC: ~3–4 weeks) | **Risk:** Med-High
- **Pros:** Soft provenance gains teeth (signed, timestamped, relay-witnessed); no own server; portable, censorship-resistant replays.
- **Cons:** Relay event-size limits force chunking design up front; keystroke cadence becomes semi-public — a privacy decision; most plumbing before the first cool moment; NDK is Dart, pulling away from CodeMirror's web-native strengths.
- **Reuses:** NDK (Dart).

## Recommended Approach

**Approach B**, chosen by the builder over the recommended A→B staging — consistent with their daily-driver conviction at D4. Build the kernel and the thin server as one project; the client is identical to Approach A, so the kernel still lands first in the build order and the first replay plays before the server is finished.

The visual reference for the replay viewer is the approved wireframe (screenshot: `/Users/peterwei/Desktop/wokspace/jamn/design-assets/gstack-sketch-replay.png`; source: `gstack-sketch-replay-viewer.html` alongside it — move into the product repo when it exists). Its locked grammar:
- Timeline: solid segments = sessions (width ∝ writing time); notched fixed-width connectors = gaps, labeled with true durations.
- Gap title-cards ("2 days later — Jun 8, 11:42 PM") as beats in the document during playback.
- Cursor as playhead; unwritten text invisible but layout-reserving (no reflow jumps).
- Header stat: "4h 12m of writing · across 9 days · 7 sessions."
- Speeds: 1×/2×/4×/skim.

**Event schema (the one-way door — design once, carefully):**
`schemaVersion, docId, eventId, authorId, deviceId, wallTime (ms epoch), seq (per-device monotonic), globalSeq? (server-assigned), origin, from, to, insert, selectionBefore?, selectionAfter?`
- **`globalSeq` is the canonical reconstruction order.** The server assigns it on accept; replay, text reconstruction, and history all order by `globalSeq` (locally-unsynced events sort after, by device `seq`). `wallTime` is **display-only metadata** — it drives gap labels and session derivation but never ordering. After a rebase, wallTimes can be locally non-monotonic; gap derivation clamps negative deltas to zero (no backwards-running title cards).
- `origin` comes from CodeMirror's `userEvent` annotation: `type | paste | undo | redo | drop | ai | unknown`. This is what makes paste a distinct replay beat (Open Q1) and AI-inserted text representable (Open Q5) without heuristics.
- `from`/`to` are **UTF-16 code-unit offsets**, matching CodeMirror document positions exactly.
- `schemaVersion` on every event — an append-only log can never be rewritten, only reinterpreted. (Scope: append-only applies to server-accepted, `globalSeq`-assigned events; locally pending events are provisional and may be rewritten by rebase before acceptance.)
- Append-only. Deletes are events, never removals. Sessions are derived (gap > threshold ⇒ new session, computed over `globalSeq` order), not stored — so the gap threshold stays tunable forever.

**Multi-device merge policy (the one real CS problem — decide before the schema freezes, not mid-build):** position-based events from two devices that diverged offline cannot be union-merged; interleaving corrupts the text. v1 policy: **single-active-writer lease per doc, at two levels.**
- *Across devices:* the server grants a per-doc edit lease with a short TTL renewed by heartbeat while the doc is focused, released on idle/close. The lease is an **optimization that makes rebase rare, not a lock**: a device may write offline without holding it, and on reconnect its pending events always funnel through sync-and-rebase — map pending events through the missed ChangeSets via CodeMirror position mapping; **fork the doc** rather than corrupt it when rebase is unsafe — concretely: fork when any pending event's `[from,to)` range intersects a range touched by the missed ChangeSets (`ChangeDesc.touchesRange`), or when a pending deletion's target text no longer matches after mapping. (Position mapping never mechanically "fails" — without an explicit trigger the fork path is unreachable and conflicting edits get silently mapped through.) Forking is ugly and honest; silent corruption is neither.
- *Within one browser:* two tabs share a deviceId and the local DB — the same divergence below the server's visibility. Single writer per doc enforced via the **Web Locks API**; additional tabs open the doc read-only (live-updating via BroadcastChannel).

**IME/composition coalescing (eng review D7, outside voice):** the recorder buffers transactions while an IME composition is active (CodeMirror marks them) and emits **one event per committed text** — pinyin/candidate intermediate states are never durable events. Event `wallTime` = commit time; composition duration shows up honestly as the pause before it. Focused tests: CJK input, autocorrect replacement, emoji picker.

**Recording latency discipline (hardened by eng review D9):** events append to an in-memory queue and flush to IndexedDB per animation frame/idle tick — never an awaited write per keystroke (that lags at typing speed) — **plus a time-based fallback: flush at least every ~250ms even when frames stall** (throttled/backgrounded tabs stop firing rAF). Mandatory flush on `visibilitychange`/`pagehide` — but these never fire on process kill or battery death, so the honest guarantee is: **"saved" never lies (indicator reflects last flushed seq), and abrupt termination loses at most the unflushed buffer, bounded by the flush interval (~250ms of typing).** Testable: assert the buffer never exceeds the flush interval under simulated throttling. Call `navigator.storage.persist()` at startup; with sync running, the eviction exposure window is unsynced events only.

**Markdown in v1 = raw markdown source with syntax highlighting** (headings/emphasis styled in place, CodeMirror markdown mode). No rendered preview pane; rendering is deferred. The replay viewer plays back the same raw-source view.

## Open Questions

1. **Paste semantics.** A paste is one event with a large `insert`. In replay it should render as a distinct visual beat (instant block appearance), not synthetic typing — also the honest choice for soft provenance. Exact treatment TBD in the viewer.
2. **Redaction.** Publishing a replay publishes deleted text and timing — things the writer may not want public (premise 1 says they own it; ownership includes redaction). What does "redact this passage from the public replay" do to the log? Likely: published bundles are derived artifacts with redaction baked in; the private log stays complete.
3. **Gap threshold.** What pause length ends a session — 5 minutes? 15? Derived sessions make this tunable; pick by feel against real recordings.
4. **Replay file size.** A long document is hundreds of thousands of events. Bundle format may need chunking/compression (events compress extremely well). Measure with real data before optimizing.
5. **AI-assisted text.** If text is ever inserted by a tool (autocomplete, Claude), is it a distinct authorId? (`origin: ai` plus premise 6's author id make this representable; policy TBD.)
6. **The right to destroy.** Redaction (Q2) covers published bundles; this is different — can the writer delete a document's *private* log entirely, including from the sync server? Premise 1 (the writer owns the log) implies yes, which tensions with append-only and with backups. Needs an answer before the server ships: likely whole-doc tombstone + server-side erasure job, with "append-only" scoped to *within* a living document, not to the writer's right to destroy one.

## Success Criteria

- **The dogfood bar (the real one):** 30 days from first usable build, the builder drafts everything non-trivial in it by choice. If they drift back to other editors, the daily-driver premise failed regardless of features.
- **The replay bar:** watching a replay of a real essay is genuinely interesting to its author — at least one "I didn't remember writing it that way" moment.
- **The share bar:** one published replay link sent to a friend; the friend watches past the first gap card without being asked to.
- **The trust bar (restated by eng review D9):** no keystroke ever shown as saved is lost, across normal use including a mid-sentence laptop-lid close. Abrupt termination (process kill, battery death, hard crash — including in throttled/background tabs) loses at most the unflushed buffer, bounded by the ~250ms fallback flush interval. `pagehide`/`visibilitychange` flushes cover every survivable exit path; the fallback timer bounds the unsurvivable ones.

## Testing (eng review D4)

**Stack:** Vitest (unit — native to the Vite client; also covers replay-core and sync logic) + Playwright (E2E — multi-context tests are how two-tabs and two-device divergence get exercised in CI). Tests land alongside each build step, never deferred.

**Coverage bar:**
- `replay-core` at **100% line + branch** — it's pure (log in → frames out), so saturation is cheap. Table-driven cases: globalSeq ordering, unsynced sort-after, gap threshold + negative-delta clamp, pause cap, skim bucket sizing + min-dwell, paste/AI beats, empty doc, paste-only doc, emoji/IME (UTF-16 offset) edges.
- **Property test:** the same event log fed through both shells' replay-core produces an identical frame sequence — makes preview/published drift mechanically impossible.
- **Non-negotiable unit tests (silent-failure paths):** the fork trigger (pending range ∩ touched range; deletion-text mismatch) and the flush bound (saved indicator reflects flushed seq; pagehide flush fires).
- **7 E2E flows (Playwright):** write→close→reopen intact; laptop-lid mid-sentence; two tabs (second read-only); two devices diverge offline → rebase-or-fork, never corrupt; replay past a gap card + backwards scrub; publish → open logged-out → identical playback; skim of a 100k-event doc completes ≈60s (perf assertion).
- Server unit tests: globalSeq monotonic and gap-free under concurrent appends; **push idempotency (same batch pushed twice → stored once)**; lease TTL/heartbeat; capability slug entropy + noindex headers; rate limit; destroy tombstone + erasure.
- Recorder tests: IME composition coalescing (CJK, autocorrect, emoji picker → one committed event each); **flush-bound assertion under simulated rAF throttling (buffer never exceeds the ~250ms fallback interval)**.

Regressions: none possible at greenfield; the REGRESSION RULE applies from the first refactor onward.

## Distribution Plan

- **v1:** web app, deployed on Fly.io/Railway-class hosting (server is a single small process + SQLite to start). The server is the canonical durable copy (browser storage is evictable, `storage.persist()` is best-effort), so **SQLite streams to object storage via Litestream from the first deploy of Step 3** — a lost volume plus an evicted browser must never mean total corpus loss. Private alpha: just the builder; accounts exist for multi-device, not for growth.
- **Privacy boundary (eng review D8, outside voice):** v1 stores private event logs in plaintext SQLite on the builder's own server — an explicit, documented trust boundary, acceptable while the only user is the operator. **Litestream replicas are encrypted at rest** (Litestream age encryption) so backups never multiply plaintext copies. Step 5 includes a one-page threat model (capability URLs, magic links, backups, deleted drafts, published redactions). **Hard gate: no second user joins before the E2EE decision is made explicitly** — keystroke cadence is behavioral data; another person's hesitations never sit readable in backups by default.
- **Published replays:** clean public URLs served by the same server; the replay viewer is a static, dependency-light page so links stay alive cheaply.
- **CI/CD:** GitHub repo + Actions deploying on merge to main. Set up via /setup-deploy once the repo exists.
- **Later:** if the format catches on, the replay bundle spec + static viewer could be open-sourced independently of the editor (the format is the network effect; the editor is the capture device).

## Next Steps

1. **Scaffold the kernel** (CC: ~1 day): Vite + React + TS + CodeMirror 6; every transaction appended as an event to IndexedDB via Dexie; write-before-saved-indicator discipline from the first commit. **Doc list is an honest doc list (eng review D11):** create, rename, delete (surface for the Q6 tombstone), and export/import archive (events as JSONL) — daily-driver table stakes, not deferred chrome.
2. **Replay viewer v0** (CC: ~1–2 days): **all playback logic lives in `replay-core` — a pure TypeScript module (event log in → timed frames out), no DOM, no framework (eng review D3).** Gap clamping, pause caps, skim coalescing, paste beats, and session derivation exist exactly once there; the in-app React viewer and the published static page are thin shells (render + controls only) around the same module, so author preview and reader playback are bit-identical by construction. Viewer features: reconstruct text from the log; cursor-as-playhead; in-session pause cap; gap title-cards; 1×/2×/4×/skim; the wireframe's timeline scrubber. **Scrub performance (eng review D5):** at replay load, replay-core keeps an in-memory document snapshot every ~1k events (one O(n) pass); jumping or scrubbing to any playhead position costs at most k event applications, never a from-zero replay — backwards scrubbing stays smooth on 100k-event docs by construction. Q4's measure-first deferral covers bundle *size* only; interactive scrub latency is solved up front. **Skim** = event coalescing, not rate scaling: bucket the log into time-weighted chunks sized so ~60s ÷ bucket-count ≥ one render frame, and play one composite diff per bucket. (Per-event playback can't beat the frame rate — 100k events at 16ms each is 27 minutes, so skim must collapse events, not accelerate them.)
3. **Minimal sync — durability first** (CC: ~1–2 days) *(pulled forward by eng review D1: Safari wipes all site storage after 7 days of inactivity, and `storage.persist()` is best-effort — real dogfood writing must never be IndexedDB-only)*: Bun + SQLite; append-only event push/pull with `globalSeq` assignment; **push is idempotent — server enforces `UNIQUE(eventId)` and silently dedups, so a retried push after a lost response never double-appends** (from eng-review failure-mode analysis; double-appended events would silently corrupt replay timing). Single-user token auth (console-printed link in dev — no email dependency yet); **Litestream replication, encrypted at rest, from the first deploy.** No lease/rebase yet — single device, divergence impossible.
4. **Dogfood checkpoint:** write one real piece, watch the replay. This is the assignment (below) — now backed by real durability.
5. **Sync hardening** (CC: ~2–3 days): the lease/rebase/fork merge policy above; magic-link auth (Resend/Postmark in prod, console-printed links in dev); second device test, including the laptop-lid divergence scenario from the trust bar. **This step also owns Open Q6:** decide and implement the destroy policy (whole-doc tombstone + server-side erasure job) before multi-device ships.
6. **Publish flow** (CC: ~2–3 days): **publish = snapshot-at-publish** — flipping the flag derives a frozen replay bundle (redactions baked in, per Open Q2) served at the public URL; continuing to write does *not* update the public replay; re-publishing explicitly replaces the bundle. Redaction question (Open Q2) gets answered here. **Public URL surface (eng review D2):** publish slugs are 128-bit random capability URLs (holding the link is the permission); `noindex, nofollow` headers by default with an explicit "listed" opt-in reserved for later; basic rate limiting on the public bundle endpoint.
7. **v1.5 candy (after dogfooding):** struggle heatmap; "watch this paragraph get born" paragraph-scoped clips; deleted-passages layer.
8. **v2 (only if still fun):** collab via CodeMirror's collab primitives or Yjs *alongside* the narrative log (per Codex: never as a replacement for it); Approach C migration if Nostr pull returns.
