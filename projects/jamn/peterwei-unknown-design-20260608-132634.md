## Problem Statement

People want place-based digital community — persistent, location-tethered conversation tied to the places they care about (neighborhoods, campuses, cafes, parks). Existing tools either flatten geography (Discord, Slack) or fail on privacy and safety (Yik Yak, Jodel) by requiring live location sharing.

The user's key insight: **decouple participation from physical presence.** Channels are placed at geographic coordinates, but anyone can join from anywhere. Geography is the organizing principle, not a tracking mechanism. This inverts the cold start problem — channels accumulate value over time regardless of who's physically nearby.

## Status Quo

Current solutions and their failures:

- **Yik Yak / Jodel** — Hyper-local anonymous posting, but died from cyberbullying + the privacy nightmare of "my harasser is in this room." Required live location.
- **Nextdoor** — Neighborhood-based, but email-thread culture, not real-time. Feels like a HOA meeting, not a community.
- **Discord / Slack** — Great real-time chat, but geography is irrelevant. You join by interest, not by place.
- **Facebook Groups** — Local groups exist but are buried in the Facebook interface and have the worst signal-to-noise ratio on the internet.

**The gap:** Nothing combines real-time chat + geography-as-organizing-principle + privacy-by-design (no live location tracking).

## Target User & Narrowest Wedge

**User type (not yet specific person):** People who are invested in the places they frequent — regulars at a coffee shop, students on a campus, residents of a neighborhood. They want to know what's happening at their places, contribute to the conversation about those places, and feel connected to the community of those places.

**Narrowest wedge:** A single campus or neighborhood. Launch with 5-10 channels at specific, real places that people already congregate around. The channels need to feel like the digital version of the corner store bulletin board — not empty, not corporate, just community chatter.

**The cold start wedge:** Channels have value even when empty because the founder (or early users) seed them with useful content — hours, events, local knowledge. The first user doesn't need another user to be online.

## Constraints

- **No live location tracking.** Ever. Users place channels, not themselves.
- **Privacy-first.** Participation doesn't reveal your physical location.
- **Must work at zero density.** A channel for a cafe has value even when nobody's sitting in the cafe right now.
- **Text-first v1.** Real-time text chat + pinned messages. No images/video/audio/file attachments. Video/audio is v2.
- **Mobile-first.** This is a phone app. People interact with places while moving through the world.

## Premises

1. **Behavior change is justified** — The pain of not having place-based digital community is real enough that people will adopt a new app. (Agreed, but unvalidated — see Assignment #1)
2. **Location is the gap** — The reason existing apps don't satisfy this need is fundamentally about location handling, not content quality or network effects. (Agreed)
3. **Real-time comms are essential** — Place-based community needs real-time interaction, not just async posts. (Agreed by founder; committed to real-time text chat in v1. Async features like pinned notes are also in v1, but the core loop is live chat.)
4. **Cold start is solvable** — Decoupling participation from physical presence means channels accumulate value independently of user density. (Agreed — this is the key insight)

## Approaches Considered

### Approach A: Place-as-Channel (CHOSEN)
Map interface with chat channels pinned to geographic locations. Anyone can create a channel at any coordinate. Anyone can join from anywhere. Text chat only in v1.

- **Effort:** S | **Risk:** Low
- **Pros:** Solves cold start (content persists), privacy-safe (no location tracking), simplest to build
- **Cons:** "Discord with a map" risk — is geography enough differentiation when participation is remote?

### Approach B: Geo-Gated Place Channels
Same as A but with soft geo-gating: full participation from anywhere, but enhanced features when verified nearby (verified local badge, location-only threads, boosted visibility).

- **Effort:** M | **Risk:** Med
- **Pros:** Geography actually matters. More defensible. Answers "why not Discord?"
- **Cons:** Geo-verification complexity. Requires some location disclosure.
- **Note:** Can be layered onto Approach A once traction exists. This is the v2 roadmap.

### Approach C: Spatial Annotation Layer (Not chosen)
Not a chat app — a map with public notes/photos/voice memos pinned to coordinates. Digital graffiti. Works at zero density but lower engagement ceiling.

- **Effort:** S | **Risk:** Low
- **Pros:** Zero cold start problem. Genuinely novel. No moderation nightmare.
- **Cons:** Lower engagement. Utility, not social. Harder to build community.
- **Note:** Elements of this (pinned notes at locations) should be incorporated into Approach A as a feature.

## Recommended Approach

**Approach A: Place-as-Channel**, with a clear path to Approach B (geo-gating) as v2.

The core product:
1. **Map view** — Browse channels pinned to real-world locations
2. **Channel view** — Real-time text chat about a specific place. Join from anywhere.
3. **Channel creation** — Drop a pin, name the place, start chatting
4. **Channel discovery** — Browse the map, search by place name, see popular/trending channels. "Nearby" channels available as opt-in when user grants ephemeral location access (used once, not stored). Default discovery is map browsing + search, not GPS-dependent.

The key differentiator from "Discord with a map" is that geography isn't just organizational — it's the **reason the community exists.** People in #cafe-lolo have something in common: they all care about Cafe Lolo. That shared relationship to a place creates natural community in a way that topic-based channels don't.

## V1 Defaults (blocking questions resolved)

- **Channel structure:** One flat channel per place (Discord VC model, not server model). No sub-channels. Threading added later when channels get busy.
- **Channel creation:** Authenticated users only. No duplicate detection in v1 — if two channels exist for the same place, users will gravitate to one. No channel deletion in v1. Creator has no special privileges beyond naming the channel.
- **Channel lifecycle:** Channels don't die. Inactive channels (no messages in 30 days) are deprioritized in discovery but remain accessible. No merging in v1.
- **Content in v1:** Plain text, emoji, links with previews, pinned messages. No images, no video, no audio, no file attachments in v1.
- **Moderation in v1:** User reporting + admin review. No algorithmic moderation. Scale this when you have a moderation problem (i.e., when you have users).

## Open Questions (post-launch)

1. **What makes a channel alive vs. dead?** How to surface active channels and bury stale ones in the map view.
2. **Discovery vs. spam** — How do you surface relevant channels without overwhelming users with every pin in a 5-mile radius?
3. **Business model** — TBD. Follow from user behavior. Do not pre-design monetization.

## Success Criteria

- **Week 1:** 5 active channels in one neighborhood/campus, each with 10+ messages/day
- **Month 1:** 50 active channels, 500 MAU, 30% D7 retention
- **Month 3:** 2 neighborhoods/campuses with organic channel creation (users creating channels without prompting)
- **PMF signal:** Users create channels the team didn't set up. Users check the app when they arrive at a place.

## Distribution Plan

- **Platform:** Mobile-first (iOS and Android via React Native or similar)
- **Distribution:** App Store + direct sharing (channel links work like Discord invites — share a link to a place's channel)
- **Viral mechanism:** Channel links shared in existing communities (group chats, Discord servers, Instagram stories)
- **Launch strategy:** Seed in one specific neighborhood or campus. Don't go wide. The product needs density in one place before it spreads.

## Tech Stack (v1 — subject to change)

- **Frontend:** React Native (iOS + Android from one codebase)
- **Backend:** Node.js with Express, WebSocket server via Socket.io
- **Database:** PostgreSQL for channels, users, messages. Redis for real-time pub/sub and presence.
- **Map:** Mapbox GL JS / Mapbox React Native SDK
- **Auth:** Phone number OTP (via Twilio) or email magic link. Anonymous reading, authenticated posting.
- **Hosting:** Fly.io or Railway (low friction, scales with usage)
- **All commodity infrastructure.** No novel technical risk.

## Dependencies

- **Map API** — Mapbox (preferred for pricing and customization over Google Maps)
- **Real-time chat infra** — Socket.io for WebSocket, PostgreSQL for message persistence
- **Auth** — Twilio for phone OTP or custom email magic link
- **No external blockers** — All required infrastructure is commodity.
