# Design: Mounted Replay Sessions and Trace-Aware A/B Assistance

## Problem Statement

Zine already records file steps, folder membership deltas, and buffered panel-focus observations, but mounting a file or folder does not yet produce a reliable, legible replay session. The controls sometimes remain empty or stale, multiple mounted histories lack an explicit merge contract, and replay-created panel state has been coupled tightly enough to the live workspace that manually closing tabs during replay caused a crash.

This is not only a replay-UI problem. Zine's product thesis is that the process trace contains useful information that the final draft discards: intent, revision rhythm, human/model contribution, abandoned directions, and the provenance of quoted or inherited ideas. That trace should improve the writer's AI assistance and eventually let a reader's AI answer questions about how a work came to be. The first product obligation is to make the trace dependable enough to dogfood and measurable enough to test whether it actually improves assistance.

## Demand Evidence

- The founder is currently the only user and has the problem directly: the present workflow is "piping text back and forth between AIs," publishing AI writing with or without an AI-slop label, or attempting to write alone.
- The founder's concrete trust objection is social and process-based: "if your AI wrote it, I'm not going to read it" because the text may not carry the writer's voice and its quality is hard to assess. The proposed counter is not an authorship badge but inspectable evidence of collaboration.
- The founder pointed to the Garry Tan and Naval Ravikant discussion about whether people should write for themselves as evidence that the concern is culturally live. This is useful problem evidence, but it is not user-demand validation for Zine.
- No outside user has yet been shown a Zine trace. Demand remains founder conviction plus public discourse, not observed adoption or willingness to pay.

## Status Quo

The client maintains an ordered mounted scope in `scope-model.ts`. A mounted folder includes descendants; a mounted file includes only itself. `App.tsx` already attempts to bootstrap replay when the attached folder becomes ready and re-bootstrap when the scope or frozen set changes. `beginReplay()` fetches folder activity, reconstructs each file's authoritative chain, filters it through the effective scope, adds structural folder deltas, and sorts the resulting entries by `steppedAtMs`.

The partial implementation has five material gaps:

1. Replay loading has no explicit `loading`, `empty`, `error`, or stale-request state, so mount changes can leave controls apparently inert or let an older async result win.
2. A timestamp sort does not state how causal per-chain order survives bad or equal clocks, and its same-time ordering depends on fetch/insertion order.
3. One folder event with several structural deltas is expanded into several UI entries even though it is one provenance Step.
4. Focus deltas exist in the protocol and storage but are skipped by replay. When focus history is absent, current panel animation has no explicit distinction between observed history and inferred staging.
5. Timeline derivation, async fetching, live editor restoration, and panel mutation remain concentrated in `App.tsx`, making replay lifecycle races hard to reason about.

## Target User & Narrowest Wedge

The first user is Peter writing real long-form drafts about consciousness and Christianity in the Zine desktop client.

The narrowest useful wedge is:

- mount one or more files or folders inside the currently attached Zine folder;
- immediately receive an accurate replay session and lit step count;
- scrub or play a deterministic interleaving without losing the live tabs used for writing and reference;
- see which panel movements were recorded and which were staged by the client;
- optionally run the same writing request in a blind final-only versus trace-aware comparison.

This wedge is intentionally writer-first. Reader-side trace interrogation, proof-of-human challenges, training on traces, and memetic/citation-graph visualization depend on trustworthy traces but are not required to test the immediate claim that trace context improves assistance.

## Constraints

- The code and feature must remain open source.
- The product is in development and almost ready for founder dogfooding; correctness for one real workflow matters more than speculative scale UI.
- Existing live panels and tabs must remain available during replay so the writer can edit elsewhere, close tabs for space, and copy material from historical views.
- Replay-created panels and tabs must be visually owned by replay with a dashed white-and-gold treatment. Existing surfaces must not be moved, duplicated, or deleted by replay.
- Recorded focus is historical observation. Inferred panel placement must never be serialized, signed, or presented as provenance.
- Focus observations do not mint Steps under `protocol/trace-provenance.md`; they ride the next structural folder Step. The stepper therefore counts provenance events, not focus cues or animation frames.
- File-chain order comes from `prev` links, never `created_at` sorting.
- Relay reads are best-effort and may be slow, partial, duplicated, malformed, or unavailable.
- No wire-format or trust-semantics change is authorized by this design.
- A/B measurements are local and private by default. No writing content or evaluation result is published without a separate explicit action.

## Premises

1. Mounting is a query lens over existing history, not a provenance event. A semantic mount change automatically creates a new replay load and never manufactures history.
2. A replay session has two truths: signed temporal history and spatial presentation. Temporal entries may be interleaved; unrecorded spatial choreography may only be a labeled, deterministic staging layer.
3. The step count represents unique provenance Steps. Recorded focus cues and per-character playback frames animate within or between those stations but do not increment the count.
4. The claim that trace context improves AI assistance should be tested with blinded pairwise comparisons, not inferred from replay delight or the founder knowing which answer used trace.
5. The immediate success criterion is better assistance for a real writer. Proof-of-human, reader-side vetting, training corpora, and meme genealogy remain downstream possibilities rather than requirements for this implementation.

## Approaches Considered

### Approach A: Patch the Current Replay

Add loading flags, a generation counter, deterministic tie-breakers, focus parsing, and safer panel bookkeeping directly around `beginReplay()` and the existing effects in `App.tsx`.

- Effort: Medium
- Risk: Medium
- Advantages: smallest diff; maximum reuse; fastest initial ship.
- Disadvantages: keeps fetching, interpretation, editor mutation, and layout orchestration coupled; makes race and lifecycle behavior difficult to test without rendering the entire app; compounds an existing `App.tsx` hotspot.
- Reuses: current `beginReplay`, `ReplayTransport`, scope helpers, chain reconstruction, and replay panel-layout helper.

### Approach B: Replay-Session Planner

Extract a pure replay-session interpretation layer and give React a thin async controller around it. The session planner emits provenance stations, recorded focus cues, and normalized trace data. A separate pure layout planner derives workspace-dependent staged instructions from that session.

- Effort: Large
- Risk: Medium
- Advantages: one semantic source for replay and trace-aware assistance; deterministic unit-testable merge; explicit ownership boundary between live workspace and replay surfaces; no protocol change.
- Disadvantages: requires a careful extraction from `App.tsx`; must avoid turning the session model into a second state store; A/B execution still needs a separate UI/controller slice.
- Reuses: `scope-model.ts`, `eventMeta`, `fetchFolderActivity`, `fetchChain`, `fetchFolderNodes`, reconstruction helpers, `panel-occupancy.ts`, `replay-panel-layout.ts`, `ReplayTransport`, and existing settings persistence patterns.

### Approach C: Parallel Replay Lanes

Render one lane per mount under a shared time cursor instead of staging file histories into editor panels.

- Effort: Extra Large
- Risk: High
- Advantages: visually honest about parallel independent histories; scales conceptually to comparison and analysis.
- Disadvantages: creates a new primary replay UI; weakens the connection to the writing workspace; does not itself solve trace-context evaluation; too large for first dogfood.
- Reuses: scope and provenance fetch/reconstruction, but little of the current panel replay UI.

## Recommended Approach

Choose Approach B. It fixes the immediate replay lifecycle while creating the reusable normalized input the A/B test needs: a deterministic, inspectable `ReplaySession` derived from the mounted lens.

Approved interaction sketch: `/tmp/zine-replay-session-mockup.png`.

### Semantic model

The planner should use data, not service classes:

```ts
type ReplaySessionStatus =
  | { kind: "idle" }
  | { kind: "loading"; scopeKey: string; requestId: number; completed: number; total?: number }
  | { kind: "ready"; session: ReplaySession }
  | { kind: "degraded"; session: ReplaySession; issues: SourceIssue[] }
  | { kind: "empty"; scopeKey: string }
  | { kind: "error"; scopeKey: string; message: string; retryable: boolean };

interface ReplaySession {
  id: string;                 // deterministic fingerprint, not a published id
  scopeKey: string;
  stations: ReplayStation[];  // unique signed provenance events
  recordedCues: RecordedReplayCue[]; // signed focus observations, never Steps
  sourceChains: SourceChainSummary[];
  issues: SourceIssue[];
}

interface ReplayStation {
  eventId: string;
  source: { kind: "file" | "folder"; path: string };
  steppedAtMs: number;
  chainIndex: number;
  fileState?: ReconstructedFileState;
  membership: FolderDelta[];  // zero or more deltas from the same folder event
}

interface RecordedReplayCue {
  observedAtMs: number;
  replayOrdinal: number;       // monotonic station coordinate used by scrub/play
  carriedByEventId: string;
  carrierOrdinal: number;
  carrierChainIndex: number;
  deltaIndex: number;
  action: "mount" | "unmount";
  path: string;
  preferredPanel?: number;
}

interface ReplayLayoutPlan {
  recorded: RecordedReplayCue[];
  staged: StagedReplayCue[];
}

interface StagedReplayCue {
  replayOrdinal: number;
  action: "mount" | "unmount" | "focus";
  path: string;
  preferredPanel?: number;
}

interface SourceIssue {
  source?: { kind: "file" | "folder"; path: string };
  code: "capped" | "unreachable" | "malformed" | "conflict" | "broken-chain";
  message: string;
  details?: { cap?: number; omittedFromId?: string; omittedToId?: string; eventId?: string };
}

interface TraceContextProjection {
  fingerprint: string;
  text: string;
  tokenEstimate: number;
  policy: TraceContextPolicy;
  includedStationIds: string[];
  omittedStationCount: number;
}
```

`ReplaySession` is normalized derived history, not prompt text or workspace presentation. Its id fingerprints the semantic scope key, planner-policy version, ordered accepted event ids, recorded cues, and a canonically sorted tuple for every issue: `(source kind, source path, code, canonical details)`. React owns only load status, the current station id, playback state, a historical display override, replay-surface ownership, and session-local layout overrides. The session planner must not call React setters, inspect workspace panels, mutate editors, or publish events. `planReplayLayout(session, startingWorkspace, sessionOverrides)` separately returns `ReplayLayoutPlan`; staged cues and workspace state never enter the session fingerprint. `TraceContextProjection` is derived lazily for a particular action, instruction, document revision, model budget, and projection policy; it is not cached as part of the session.

### Fetch and lifecycle

1. Compute the semantic key from attached folder id, ordered `scopeKey(scope)`, and sorted frozen boundaries.
2. On a key change, stop playback, clear the historical display override, and enter `loading` immediately so the controls visibly respond.
3. Increment a monotonic request id. Explicit file mounts fetch those paths directly. Folder mounts use folder activity to discover every historical file path under the mounted prefix, including paths no longer in current membership. Apply frozen boundaries to each discovered historical path; a direct child mount overrides an exclusion inherited from a broader mount exactly as `pathInEffectiveScopes` does today. Structural rename stations are included when either `fromPath` or `toPath` intersects the effective scope.
4. Bound discovery to 2,000 activity events for the first dogfood implementation. A result that hits the cap is `degraded`, never exact, and displays `2,000+ events loaded`. Fetch relevant chains through a concurrency-four queue. Abort queued work on a newer request; where an active relay request cannot accept `AbortSignal`, let it finish but discard its result.
5. Reuse an in-memory chain cache keyed by `(folderId, path, resolvedHeadEventId)` and a folder-chain cache keyed by `(folderId, folderHeadEventId)`. A changed local file node id or folder head invalidates only the affected entry. No persisted cache is introduced.
6. Normalize, validate, and build the session through pure functions. Commit only if both request id and semantic key still match.
7. Resolve to `ready` only when every requested source completed and discovery did not hit its cap. Resolve to `degraded` with per-source failures/caps when at least one valid station exists. Resolve to `empty` only when every requested source completed successfully and all are genuinely empty. Zero usable stations plus any failed, capped, conflicting, or malformed source is `error` carrying those source issues. Only network and timeout issues are retryable. The fixed `capped` condition is non-retryable because the same query would hit the same bound; loading beyond it requires the separately deferred pagination/query strategy. Invalid configuration/data/planner failures are also non-retryable. Malformed individual events degrade a session that otherwise has valid stations rather than crashing it.
8. New local Steps replan from cached normalized chains through the same merge function, rather than using a second timestamp-sort implementation.

Rapid mount changes are expected behavior. A stale response is not an error and must not flash an error state.

The sidebar renders `loading x/y`, `N steps`, `N loaded · M unavailable`, `no history`, or `could not load · Retry`. It never carries an old count across a semantic key change and never calls a partial count exact.

### Input validation and event identity

- Canonical identity is the signed event id. Deduplicate relay responses by id before grouping or chain walking.
- Reuse the client's existing event/signature validation boundary. Reject an event whose bytes do not hash to its claimed id, whose kind/discriminator is wrong for the source, or whose required folder/path/`prev` metadata is malformed.
- If two relay payloads claim one id with different bytes, reject both payloads for that id and add a source issue; do not choose by arrival order.
- Normalize time as a finite non-negative content `steppedAt`, else a finite non-negative `created_at * 1000`, else `0`. Chain order remains authoritative even when normalized times are missing or regress.
- A `prev` cycle, missing interior predecessor, or multiple uncited heads makes that source degraded. The planner may retain the single verified contiguous segment ending at the resolved head, but it reports the omitted portion.

### Deterministic interleaving

Each file chain and the folder chain is first reconstructed in authoritative `prev` order. The global timeline is then a deterministic k-way merge:

1. Only the next unconsumed event from each source chain is eligible.
2. Choose the eligible event with the lowest tuple `(steppedAtMs, sourceKindRank, path, eventId)`, where folder rank is `0` and file rank is `1`.
3. Advance only that source and repeat.

This preserves causal order even when a later event has an earlier or equal wall-clock timestamp. It also makes equal-time independent histories deterministic across fetch order and reloads. A resulting time axis may therefore be non-monotonic; the existing scrubber must use its documented index-linear fallback in that case. A folder node with multiple structural deltas produces one `ReplayStation` containing all of them.

The currently attached folder is the provenance boundary for the first implementation. Multiple mounted paths may be interleaved inside it. Mounting histories from several unrelated folder identities is a separate feature because it requires a cross-folder query and context policy, not a different sorting rule.

### Recorded focus and staged layout

- Parse `focus` deltas from the folder chain and retain both `observedAtMs` from the delta and `carriedByEventId` from the structural folder Step that signed it.
- After station merge, compute the monotonic prefix-time envelope `T[i] = max(stations[0..i].steppedAtMs)`. A recorded cue's `replayOrdinal` is the first station ordinal whose envelope reaches `observedAtMs`; a cue earlier than `T[0]` anchors to ordinal `0`, and one later than the final envelope anchors to the live boundary after the final station. `carrierOrdinal` records where its signing folder Step landed.
- Playback and scrub use only the monotonic tuple `(replayOrdinal, observedAtMs, carrierChainIndex, deltaIndex, opRank)`, where recorded `opRank` is `unmount = 0`, `mount = 1`. Observation time refines ordering inside an ordinal but never determines whether an already-applied cue disappears.
- Scrubbing to station ordinal S rebuilds layout from the session start using every cue with `replayOrdinal <= S`. The UI tooltip says `observed at … · checkpointed at Step N` when observation and carrier differ, so it does not imply the observation was already signed at the earlier instant.
- A recorded cue with an invalid timestamp anchors to its carrier ordinal and is marked degraded.
- A focus cue never becomes a stepper station.
- Ignore recorded cues outside the effective mounted scope for replay staging. They remain available in the raw folder trace.
- `ReplaySession.recordedCues` contains only recorded history. `planReplayLayout(session, startingWorkspace, sessionOverrides)` derives staged cues when no usable recorded cue exists for a station that must display a file. Staged `opRank` is `unmount = 0`, `mount = 1`, `focus = 2`, and every staged cue applies after recorded cues at the same ordinal.
- The layout planner uses explicit mount order and current available panel capacity. Scrubbing recomputes staged cues from the same starting workspace and closed/moved session overrides, so seeking is deterministic.
- Every rendered cue carries `recorded` or `staged`. The UI must expose that distinction in a tooltip or compact badge and accessibility label, not color alone.
- Staged cues exist only in memory. They are never passed to `bufferFocus`, included in a folder Step, signed, or published.

### Live workspace ownership

Replay must operate as a reversible overlay:

- Capture the identity of panels/tabs that pre-exist replay. Do not capture only array indices; manual closes and collapses invalidate them.
- If a replay path is already open, reuse that surface in place and apply only the transient historical/frozen indicator. Do not duplicate or relocate it.
- If a replay path is not open and needs to be shown, create a replay-owned surface with the dashed white-and-gold frame.
- Historical content is a read-only display override, `historicalByPath[path]`, layered over canonical `files[path]`; replay must stop writing historical snapshots into the live `files` state. The editor renders `historicalByPath[path] ?? files[path]`, and the existing replay-active edit gate rejects edits only for the historically displayed path.
- External/MCP/local-head updates continue to update canonical `files[path]` while a historical override is visible. Leaving that station simply removes the override and reveals the latest canonical state, so there is no captured snapshot to restore over newer work.
- A user may close or move a replay-owned surface. Closing suppresses it for the rest of that playback session; later animation must not recreate it immediately. Moving changes its placement for the rest of the session.
- A user may close any pre-existing tab. Replay must treat the resulting workspace as authoritative and continue without dereferencing a missing panel.
- On stop/end/scope change, clear historical overrides and remove only surviving replay-owned surfaces. Never reconstruct the entire pre-replay layout, because that would undo deliberate user changes made during replay.
- All panel transforms use functional updates and path/ownership identity. They must tolerate zero panels, collapsed panels, missing active tabs, and repeated close/end calls.

### Live append and cursor identity

- The current position is stored as `currentStationId`, not only an array index. UI indices are derived after each plan.
- A newly observed event is deduplicated by id, inserted into its normalized source chain, and causes a complete pure replan from cached inputs. Folder focus cues and source completeness are refreshed in the same operation.
- If the user is parked at live, the cursor follows the replanned final station. If the user is historical, the cursor remains on the same event id even if a late event inserts before it and changes its numeric index.
- If the current station disappears because a source becomes invalid, fall back to its closest surviving predecessor in the same chain, then the closest prior global station, and surface a degraded notice.
- Replanning produces a new session fingerprint. Trace-context projections are invalidated by that fingerprint and rebuilt only for the next model request.

### Trace context projection

`projectTraceContext(session, input)` accepts `{ activePath, action, instructionHash, documentRevisionHash, modelConfigFingerprint, inputCeiling, outputReserve, policy }` and creates process-only bounded text. The identical active-document snapshot and user instruction are assembled outside this function and shared by both A/B arms, avoiding duplicate active-document bytes in the trace arm. The projection does not mutate `ReplaySession`. It may include:

- final-state excerpts from other mounted reference files, but never the active document snapshot;
- ordered step summaries and timestamps;
- changed spans or compact deltas;
- human/model attribution where available, labeled as asserted or verified according to the protocol;
- citations, prompts, and action metadata already carried by the trace;
- recorded focus summaries only when useful to the request;
- no staged layout cues.

The first-cut policy is fixed rather than left to implementation judgment:

- Extend and Stir are the only eligible A/B actions.
- Both arms reserve the same model context window and output allowance and receive the identical instruction and current document bytes. If the current document plus fixed prompt already exceeds the input ceiling, the comparison is ineligible rather than truncating the two arms differently.
- The trace arm may use at most the remaining input budget, capped at the lower of 8,192 estimated tokens or 35% of the total input ceiling.
- Token estimation is versioned and injected by the model adapter. Use the provider's exact tokenizer when available; otherwise use safe fallback estimator `utf8-safe-v1 = UTF-8 byte length` tokens. Before generation, use a provider preflight count when available. On a provider context-length rejection, deterministically remove the lowest-priority whole entry and retry at most three times; if it still fails, cancel both-arm presentation and show an ineligible/error state rather than comparing unequal requests. The estimator id and any shrink count are part of the projection fingerprint/debug metadata.
- Every rank is fixed in policy version 1. `activePathRank`: active path `0`, every other path `1`. `roleRank`: prompt `0`, inline citation `1`, reply citation `2`, content citation `3`, tag citation `4`, live citation `5`, unknown citation `6`. Selection class rank: prompt/citation `0`, attribution `1`, changed span `2`, reference final-state excerpt `3`, genesis `4`. Render `entryKindRank`: genesis `0`, prompt `1`, citation `2`, attribution `3`, changed span `4`, reference final-state excerpt `5`, metadata stub `6`.
- Candidate entries have stable keys. Prompts/citations sort by `(activePathRank, roleRank, station ordinal descending, event id, edge index)`; attribution claims sort by `(station ordinal descending, path, run start)`; changed spans sort by `(station ordinal descending, event id, delta index)`; reference excerpts sort by `(station ordinal descending, path, event id)`; genesis sorts last. Selection walks class rank, then the stated class tuple. Selected entries render back into `(station ordinal, entryKindRank, stable key)` order.
- Entries are atomic. Never truncate prose, prompt, citation, or changed-span text mid-entry. If the next entry does not fit, omit it and continue to later entries that do fit. An individually oversized high-priority entry is represented only by a deterministic metadata stub containing kind, event id, UTF-8 length, and content hash. The projection records omitted counts by kind.
- If prompts/citations alone fill the allowance, lower-priority classes receive no space; this is visible in omission metadata rather than hidden by ad hoc truncation. Do not call another model to summarize and do not include staged cues.
- Every projection records its policy and estimator versions, budget, estimated tokens, included station ids, omitted counts, and a fingerprint of `(session id, active path, action, instruction hash, document revision hash, model/config fingerprint, input ceiling, output reserve, policy version, estimator id, shrink count)`. A debug inspector shows this metadata and exact local projection text.

### Blind A/B assistance

A settings toggle makes the mode available to every user but defaults off because it doubles model calls, latency, and cost. Enabling it explicitly discloses both the doubled provider cost and the local trial retention policy.

When enabled for an eligible writing action:

1. Freeze the user instruction, active document snapshot, canonical document head/revision hash, model, model settings, and available random seed. Apply the same system instruction to both arms: do not mention hidden context, provenance, or process unless the user's request explicitly asks for it; process-analysis requests are not eligible for blinded comparison.
2. Build two requests: final/current context only, and final/current context plus the bounded trace context.
3. Execute both without exposing arm identity. Randomize left/right placement independently per trial.
4. Present both outputs without applying either to the document. Before reveal, the user chooses left, right, or neither, may tag the reason (`voice`, `intent`, `usefulness`, `accuracy`, `other`), and guesses `left used trace`, `right used trace`, or `cannot tell`. The guess measures whether blindness held; it does not affect the choice.
5. Reveal which arm was trace-aware only after both the preference and arm guess are locked.
6. Applying the chosen output uses the existing attribution and stepping path only while the active document's head/revision hash still matches the frozen trial hash. If the document changed while either call was running or while the chooser was open, disable Apply, keep Copy available, and offer `Run again on current draft`; never silently rebase generated text. The rejected output never enters the document trace.

Local evaluation records are stored in IndexedDB for 30 days by default and contain trial id, time, user instruction, frozen document revision hash, both raw outputs, model/config fingerprint, replay-session and projection fingerprints, projection debug metadata, randomized arm mapping, preference, arm guess, reason tags, latency, and response hashes. They do not persist the full source trace or rendered trace projection; those remain inspectable during the live trial. Users can delete all trials immediately. Export and any publication are separate explicit actions and are not part of this implementation.

Because independent model calls may still vary despite matching settings, the app must not over-interpret one comparison. The dogfood assignment uses repeated randomized trials across real writing sessions, reports wins/losses/ties and an exact binomial confidence interval for non-tied choices, and separately reports arm-guess accuracy.

### Implementation slices

1. **Replay derivation:** pure station/recorded-cue planner, deterministic merge, unique Step counting, and unit tests.
2. **Replay lifecycle:** explicit load state, stale-result suppression, retry, and follow-newest through the planner.
3. **Surface ownership:** separate pure layout planner, existing-tab preservation, replay-owned staging, recorded/staged labels, manual close/move behavior, and cleanup.
4. **Trace context projection:** bounded deterministic context with debug inspection.
5. **A/B mode:** settings toggle, dual-run blind chooser, local outcomes, and chosen-output application.

Slices 1-3 are the replay reliability gate for dogfooding. Slices 4-5 test the product thesis. They may land as separate commits or pull requests but should share the same `ReplaySession` contract.

## Open Questions

There are no implementation-blocking product decisions left for the first cut. The following are explicitly deferred:

1. Settle and Reply eligibility. The first cut supports Extend and Stir only.
2. Permanent adoption of replay-owned tabs. Moving is session-local; a future Pin action requires its own interaction design.
3. Evaluation publication or provenance. Trials remain private/local; export and publication require a separate consent and schema design.
4. Cross-folder replay. The first cut interleaves mounts only inside the currently attached folder identity.
5. Relay pagination beyond the 2,000-event dogfood cap. Hitting the cap is visible degradation, not silent truncation.

## Success Criteria

### Replay correctness

- Mounting or unmounting any file/folder immediately transitions controls to `loading`, then shows the unique Step count when complete, a visibly partial count plus source issues when degraded, or an explicit empty/error state.
- Rapidly applying at least ten mount changes cannot let an older result replace the newest scope.
- Three mounted file chains with equal and non-monotonic timestamps replay in the same deterministic order across repeated loads while preserving each chain's `prev` order.
- Moving forward through non-monotonic stations never removes a cue already applied at an earlier replay ordinal; backward scrub reconstructs the identical earlier occupancy.
- A folder event carrying multiple membership deltas increments the stepper once.
- Recorded focus cues and staged cues are visually and accessibly distinguishable; staged cues never appear in persisted focus buffers or published events.
- Existing tabs are never relocated or deleted by replay. Closing existing and replay-created tabs during scrub/play does not crash, resurrect closed surfaces, or corrupt the active panel.
- Ending replay clears historical display overrides, reveals the latest canonical live content, and removes only replay-owned surfaces that still exist.
- Relay timeout, partial chain failure, malformed event content, empty history, zero panels, and repeated end/retry operations all have covered and recoverable behavior.
- Zero stations plus any failed source is an error, never an empty-success state; degraded-session fingerprints differ when the failing source or omitted range differs.

### Product signal

- Complete at least 20 blinded comparisons while writing the consciousness and Christianity drafts, using both eligible actions, Extend and Stir.
- Record a choice or `neither` plus an arm guess before reveal for every trial; verify left/right randomization is approximately balanced and report arm-guess accuracy.
- Changing the document after a trial starts disables Apply and offers rerun/copy without silently rebasing the result.
- Projection tests cover exact-tokenizer and fallback-estimator paths, deterministic packing, prompts/citations exhausting the allowance, oversized-entry stubs, and fingerprint changes for document revisions and policy inputs.
- Treat a trace-aware win rate of at least 60% of non-tied comparisons, with reason tags concentrated on `voice`, `intent`, or `usefulness`, as an exploratory directional signal only. Always report wins, losses, ties, sample size, and the exact binomial confidence interval; do not claim statistical proof from 20 trials.
- If it does not meet that bar, retain the trial data and inspect context projections before expanding replay, proof-of-human, or reader-facing features.

## Reviewer Concerns

The third and final allowed adversarial pass scored the document 8/10 and raised the five items below. Their resolutions are incorporated above, but there was no fourth independent pass to verify them. The engineering review must confirm these contracts before implementation:

1. **Process-only context:** the active document is explicitly excluded from `TraceContextProjection`; only other mounted reference-file excerpts may appear there.
2. **Complete deterministic ranks:** recorded/staged operation ranks plus selection, role, and render rank tables are fixed in policy version 1.
3. **Safe budget enforcement:** the fallback counts one token per UTF-8 byte and the provider path uses deterministic whole-entry shrink-and-retry on context-length rejection.
4. **History/presentation separation:** `ReplaySession` contains recorded cues only; `ReplayLayoutPlan` derives staged cues from the starting workspace and session overrides.
5. **Cap retry semantics:** hitting the fixed 2,000-event bound is visible non-retryable degradation; an actual retry requires a future pagination or changed-query strategy.

## Distribution Plan

This design introduces no new artifact. It ships in the existing open-source React/Tauri client and uses the existing source distribution and client build path. Before inviting outside dogfooders, the project needs a separate release/distribution pass covering signed Tauri builds for supported platforms and installation instructions.
