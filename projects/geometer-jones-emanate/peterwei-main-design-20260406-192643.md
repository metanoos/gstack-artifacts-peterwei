## Problem Statement

Obsidian users, especially those using LLMs to generate and modify vault content, lack version control. Edits are destructive — there's no undo history, no diff view, no rollback. Meanwhile, the Nostr protocol provides a built-in content-addressed event chain where every edit is an immutable event pointing to its predecessor. Emanate bridges these two systems by using Nostr's event chain as a version control layer for Obsidian, with the additional benefit of making vault knowledge publishable and networked.

## Demand Evidence

Founder-driven. No external demand evidence yet. The founder identified the opportunity through a structural insight: both Obsidian (wikilinks) and Nostr (event tags) use bidirectional link graphs. Bridging them creates a unified knowledge graph where personal notes become part of a networked system.

The strongest demand signal is hypothetical: every Obsidian user who uses LLMs to modify their vault will eventually need version control. The pain is inevitable but unvalidated.

## Status Quo

Nostr long-form writing tools:
- **Habla.news** — Medium-like long-form Nostr platform
- **blogstack.io** — Blogging with markdown for Nostr
- **yakihonne.com** — Censorship-resistant content media
- **notebin.org** — Markdown + code highlighting

Obsidian-to-Nostr:
- **Obsidian Nostr Writer** — Free plugin, publish from Obsidian to Nostr. Simple, does one thing.

No existing tool combines version control, bidirectional link resolution, and Nostr publishing in a single Obsidian plugin.

## Target User & Narrowest Wedge

Target: Vibe coders using LLMs to automate Obsidian wiki creation who need version control for AI-generated edits.

Wedge (v1): Version control with visual diffs and one-click rollback. Every edit creates an immutable event chain. Users can see what changed and undo LLM mistakes.

Expansion (v2): Networked knowledge. Wikilinks resolve across vaults. Publishing to Nostr. Social features.

## Constraints

- Must work as an Obsidian plugin (desktop-only; `manifest.json` should set `isDesktopOnly: true` due to sql.js WASM requirements)
- No self-hosted server component. Uses public Nostr relays for sync/sharing, not primary storage.
- Local-first persistence via SQLite (sql.js WASM). Relays are the sync layer, not the source of truth.
- NIP-49 encrypted key storage for security
- Nostr identity required only when publishing (Stage 2). Stage 1 version chains work locally.

## Approaches Considered

### Approach A: AI-Vault Safety Net (not chosen)
Pivot framing entirely. Not a Nostr client — a vault safety plugin. Nostr as invisible infrastructure. Users don't need to know what Nostr is. Lowest risk, clearest demand signal, but loses the social/networked knowledge angle.

### Approach B: Minimal Graph Layer (not chosen)
Strip to wikilink resolution + version history + publish. Cut DMs, feed, AI categorization. Focused but still requires Nostr adoption.

### Approach C: Full Vision, Staged Rollout (chosen)
Keep the full Emanate feature set but lead with version control as the wedge. Publishing is v1. Networked knowledge (resolving links across vaults) is v2. DMs, feed reader, AI categorization are staged features. Highest option value but requires discipline to not build everything at once.

## Recommended Approach

Approach C with strict staging. Only Stage 1 is designed in detail. Stages 2+ are directional, not specified.

### Stage 1 — Version Control Wedge (build and ship this now)

**Scope:** Event-chained version history with visual diffs and one-click rollback. Purely local via sql.js. No Nostr identity required.

**Data model:**

Each version snapshot is stored as a row in the local SQLite `event_versions` table:

```
event_versions:
  id          INTEGER PRIMARY KEY
  note_path   TEXT NOT NULL         -- vault-relative path
  content     TEXT NOT NULL         -- full note content at this version
  created_at  INTEGER NOT NULL     -- unix timestamp ms
  parent_id   INTEGER REFERENCES event_versions(id)  -- previous version
  source      TEXT DEFAULT 'user'   -- 'user' | 'llm' | 'import'
  checksum    TEXT NOT NULL         -- sha256 of content (for dedup)
```

When a note is published to Nostr (Stage 2), the corresponding Nostr event ID is stored alongside the version row. During Stage 1, the `nostr_event_id` column is NULL — versioning is local-only.

**Version chain algorithm:**
- Trigger: on-save (after Obsidian's `modify` event fires), debounced by configurable delay (default: 30 seconds, range: 5s–300s)
- Dedup: if checksum matches previous version, skip (no-op edit)
- Chain: each new version row gets `parent_id = previous version's id`
- LLM rapid-fire: if >10 versions created within 60 seconds for the same note, auto-throttle to batch every 60s (compress intermediate versions into a single diff)

**Auto-snapshot:**
- Timer-based (configurable, default 60s)
- Captures current state of active note
- Only creates version if content actually changed (checksum comparison)

**Diff rendering:**
- Uses the `diff` library (already in dependencies) for line-level diff
- UI: side-by-side or inline view in a modal, with colored additions/deletions
- Navigation: version list (chronological), click to view diff vs. current

**Rollback:**
- One-click: replace current note content with selected version's content
- Creates a new version row (rollback is not destructive — it's just another edit)
- Source tagged as 'user'

**First-run experience:**
- On install, prompt to import existing vault: scan all .md files, create initial version for each
- Background batch (max 100 notes/minute to avoid UI jank)
- User can skip import and start fresh

**Failure modes:**
- sql.js init failure: fall back to in-memory mode, warn user, persist on next successful init
- Storage quota: warn when database exceeds 100MB, offer to prune versions older than N days
- Corrupted chain: if parent_id references missing row, treat as root version

### Stage 2 — Publishing & Link Resolution (design after Stage 1 ships)

Directional scope:
- Publish notes as Nostr kind 1 (short-form) / kind 30023 (long-form) events
- Wikilink → Nostr event resolution algorithm
- NIP-19 entity resolver
- NIP-49 keypair management + relay config
- Requires Nostr identity — this is where relay onboarding begins

Wikilink resolution algorithm (to be specified in Stage 2 design):
- `[[Note Title]]` → look up note_path in event_versions table → find latest published event_id → create `["e", event_id]` tag
- Unresolved links (note not yet published) → `["t", "topic-slug"]` tag
- Ambiguous names → disambiguate by folder path or let user choose

## Open Questions

- Will Obsidian users care about Nostr? Or is Nostr just infrastructure? (Stage 2 question — Stage 1 doesn't require Nostr knowledge)
- Is version control alone enough of a wedge, or do users need the publishing angle immediately?
- What's the distribution strategy? Obsidian plugin directory is primary. Nostr community is secondary.
- Offline editing: Stage 1 is fully local (sql.js), so offline works. Relay sync (Stage 2) needs a queue-and-flush strategy.
- LLM rapid-fire edits: addressed by auto-throttle (batch every 60s when >10 versions in 60s).

## Technical Risks

- **sql.js WASM + Obsidian Electron compatibility:** Spike task needed before Stage 1. Fallback: IndexedDB via Obsidian's `localStorage` API.
- **Bundle size:** sql.js WASM (~660KB) + diff library. Monitor. Obsidian plugin size guidance is ~5MB max.
- **Undocumented dependencies:** `ws` and `ngeohash` are in package.json. `ws` is for relay WebSocket communication (Stage 2). `ngeohash` is for location features (future). Audit and remove anything not needed for Stage 1 to keep bundle lean.

## Success Criteria

- 10 active users within 30 days of launch
- Users creating version chains (not just installing and forgetting)
- At least one user rolling back an LLM edit (version control proving its value)
- At least one user publishing a note to Nostr via the plugin
