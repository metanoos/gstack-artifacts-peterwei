## Problem Statement

Classical Chinese (wenyanwen) reading is underserved by existing language learning apps. Pleco, Du Chinese, HelloChinese, and Skritter all focus on modern Mandarin. Classical Chinese is treated as an academic specialty, not a consumer product. Da Xue combines authentic classical texts with an AI-guided reading loop and character reference tools into a single curriculum-first reading experience.

## Demand Evidence

Users are actively using Da Xue. Not yet paying. The product combines three features (authentic texts, AI translation feedback loop, character explosion with flashcards), but usage distribution across features is unknown. No behavioral demand evidence yet (no "called me when it broke" stories, no payment willingness data).

## Status Quo

Competing apps exist but miss the classical Chinese reading niche specifically. Existing apps focus on modern Mandarin vocabulary acquisition (flashcards, spaced repetition) rather than guided reading comprehension of authentic literary texts. The workaround for serious classical Chinese learners involves physical textbooks, dictionary lookup in Pleco, and manual character study. Da Xue collapses that multi-tool workflow into a single guided experience.

## Target User & Narrowest Wedge

The target user is someone studying classical Chinese who wants to read real texts (Lunyu, Mengzi, Sanguo Yanyi) rather than study vocabulary in isolation. The wedge is the AI translation feedback loop: learner writes a translation, AI gives feedback, they iterate. Everything else (character explosion, flashcards, content catalog) supports that loop.

## Constraints

- Flutter frontend, Go API backend, bundled JSON content
- Optional GLM/z.ai API for AI features (translation feedback, guided chat)
- No usage analytics infrastructure exists today
- Revenue model undefined

## Approaches Considered

- Summary: Ship a single analytics event this week. Ship `screen_view` tracking across all routes. Ship `translation_submitted` tracking for the AI reading loop. This answers the first question: which features do users actually touch?
- Effort: S (single event) to M (full screen tracking)
- Risk: Low
- Pros:
  - Answers the most important unknown (which features matter)

  - Enables data-driven decisions for all future work
- Cons:
  - No new user-facing features
  - Go backend has no database today; event storage requires introducing a persistence mechanism

## Recommended Approach

**Approach A: Analytics MVP. Ship one event this week. Expand next week.**

Rationale: You have users but no data on what they actually do in the app. The current diff adds content and enriches character data (fine, keep doing that). The parallel priority is instrumenting the product so every subsequent decision is evidence-based.

The approaches are **sequential, not alternatives.** Phase 1 (A) produces data. Phase 2 is either B or content expansion, decided by that data.

### Engineering Design

**Identity strategy:** Generate a random UUID on first app launch. Persist in Flutter local storage (`SharedPreferences`). Send with every event. No auth, no PII. This is an anonymous device ID sufficient for cohort analysis.

**Event schema (common envelope):**
```
{
  "device_id":  "uuid-string",      // anonymous, generated on first launch
  "session_id": "uuid-string",      // generated on each app open
  "event":      "screen_view",      // event name
  "ts":         "ISO-8601",         // client timestamp
  "payload":    { ... }             // event-specific fields
}
```

**Events to ship (ordered by priority):**

| Priority | Event | Payload | Why |
|----------|-------|---------|-----|
| 1 | `screen_view` | `{"route": "/library"}` | Which features get touched |
| 2 | `translation_submitted` | `{"book": "lunyu", "chapter": "001", "segment_index": 5}` | AI loop engagement |
| 3 | `session_start` / `session_end` | `{"duration_s": 120}` | Session length, return visits |
| 4 | `character_viewed` | `{"character": "学"}` | Character reference usage |
| 5 | `flashcard_reviewed` | `{"character": "学", "correct": true}` | Flashcard engagement |

Ship priorities 1-2 this week. Add 3-5 next week.

**Storage:** Server-side append-only JSONL file. One event per line. `POST /api/v1/events` writes to `data/events.jsonl`. No database. Query with `jq`, `grep`, or a one-off Python script. This is enough for weeks 1-4. Upgrade to SQLite or PostHog when JSONL gets unwieldy (likely >100K events).

**Offline behavior:** Drop events when offline. No local queueing for v1. The data is analytics, not user data. A few lost sessions won't skew cohort conclusions.

**Flutter instrumentation:** Use a `NavigatorObserver` subclass to auto-track route changes. Add `AnalyticsObserver` to `MaterialApp.observers`. This fires `screen_view` on every navigation without touching individual screen widgets. For non-navigation events (`translation_submitted`, `character_viewed`), add calls at the existing interaction points.

**Retention definition:** A user is considered retained on day N if they trigger any event on calendar day N after their first session. Day 1 = the calendar day after first session.

**Data handling:** Only anonymous device IDs and interaction events. No PII. No user accounts. Events stored server-side in JSONL. Delete by removing the file. This is sufficient for the pre-revenue stage.

### Relationship to ongoing content work

The current diff (Sanguo Yanyi chapters, character index enrichment) is orthogonal to analytics. Keep shipping content. Analytics runs in parallel. The two converge when analytics data reveals which texts and features to prioritize next.

## Open Questions

- What is the target market size for classical Chinese reading?
- Is classical Chinese the ceiling, or is it the wedge into broader Chinese literacy?
- Do users flow between reading, character reference, and flashcards in a single session, or do they use them independently?

## Success Criteria

- **This week:** `screen_view` events landing in `data/events.jsonl` from production traffic. Verify with: `cat data/events.jsonl | jq 'select(.event=="screen_view")' | wc -l`
- **Week 2:** `translation_submitted` events also landing. Can run: `jq -s 'group_by(.device_id) | map({device: .[0].device_id, screens: length}) | sort_by(.screens) | reverse | .[:10]' data/events.jsonl` to see top users by engagement.
- **Week 4:** Enough data to compare retention of AI-loop users vs character-browser-only users. Decision: proceed to Approach B (deepen AI loop) or expand content breadth.

## Dependencies

- None. Analytics can be added to the existing stack without blocking other work.
- Go backend has no database today. JSONL file storage avoids introducing one.
- Flutter app has no analytics wiring today. `NavigatorObserver` is the lowest-touch entry point.

## The Assignment

**Ship `screen_view` tracking this week.** One event. One endpoint. One JSONL file. That's it. Don't build the full analytics suite. Just prove you can see which screens your users visit. Then look at the data yourself. You'll learn more from 50 real events than from any amount of feature planning.
